# Future Shift Advisory — Admin Data, Credentials & Calendar Plan

**Source:** owner decisions (this session) on how to operate the platform when external APIs are unavailable, partial, or not yet provisioned — plus the live findings from the Stats NZ Aotearoa Data Explorer (ADE) integration test.
**Track scope:** make the platform's **advisory functionality fully operable without depending on any external API** (manual entry / file import / built-in features as governed fallbacks), while keeping APIs an *optional, governed* enhancement. The defined exceptions with no manual equivalent — real **payment collection** and real **AI generation** — run in explicit degraded modes (sandbox/no-collection; `FakeAiClient`) and are labelled `api_required`. Three cohesive capabilities: (1) governed **manual reference-data entry** with recurring admin dashboard tasks; (2) an encrypted **admin credentials vault** + **go-live readiness panel**; (3) a **built-in advisor calendar** with third-party sync demoted to optional.
**Predecessors:** Phases 1–4 + NPO + Codex feature batch, green on `featureApp`/`main`. Builds on existing infra: `KeyEnvelope`, the governed Learning Update Queue (Layers 12/13), `IntegrationHealthController`, `meetings` + `CalendarSync`, `Permission`/`roleMatrix`, `AuditWriter`, RLS.
**Work orders:** **WO-A01 … WO-A06** (admin-data/credentials namespace).
**Sequencing:** A01 (vault) → A02 (go-live panel) → A03 (reference data) → A04 (dashboard tasks) → A05 (Stats NZ ADE live + manual fallback) → A06 (calendar). Each its own green commit on `featureApp`; `main` fast-forwards only when green (suite needs `-d memory_limit=2048M`).

## 1. Non-negotiable principles (binding)

- **Secrets never in plaintext, never in the DB unencrypted, never to the frontend, never in logs.** Admin-managed credentials are **`KeyEnvelope`-encrypted at rest** (same pattern as the calendar/accounting OAuth tokens already stored). The plaintext is **only** decrypted at the point of use inside the integration client. The UI shows **masked** values only (`····3b03`); the value is never selected into Inertia props or returned by any endpoint. **Extend the `Redactor` with key-name-based secret scrubbing** — the current Redactor only masks email/bank/IRD/phone *values*; add redaction keyed on field/key names matching `api_key`, `secret`, `token`, `client_secret`, `subscription_key`, `password`, `key` so any credential value is scrubbed from logs/audit context regardless of where it appears. **This must also cover `HealthRecorder`** — it writes raw `endpoint` + `error_payload` to `integration_calls`, where a URL query secret (`?subscription-key=…`) or an echoed upstream auth header would otherwise persist; redact the endpoint query string + error payload before that insert.
- **Credential mutation (set/rotate/revoke) and go-live are super-admin only + a FORCED FRESH step-up + audited** (who changed which integration credential/activation, when — **never the value**). New `credential.manage` permission (not overloaded). ⚠️ The existing `mfa` middleware only checks an *already-verified* session, and step-up is risk-based and same-device-exempt — so these routes need a **dedicated "fresh step-up" middleware** that **re-challenges MFA now** (verified within the last few minutes, else re-prompt), regardless of session/device risk.
- **Resolution order (status `active`/`revoked`; revoked is terminal):** credentials have two states — **`active`** (set or rotated) and **`revoked`**. The resolver: **if a DB row exists** → return the value when `active`, return **absent when `revoked`** (terminal — it does **NOT** fall through to `.env`, so in-app revoke actually stops the integration even if env still holds a key); **only when no DB row exists at all** does it fall back to `config`/`.env`. So dev keeps using `.env`; once an admin manages a credential in-app, the DB row is authoritative. **For `managed_via=vault` credentials, no code path reads a raw key except the resolver; `managed_via=environment` (mail delivery) is exempt** (it reads Laravel's mail config by design).
- **"Live" is a DB-backed activation, not a runtime env flip — and *every* live check uses it.** `FEATURE_*_LIVE` are `env`/`config` (not writable at runtime). Introduce an **`integration_activations` DB state** + an **activation resolver**: `isLive(integration_key)` = the DB activation if set, else the `config` flag, **AND `false` whenever `wiring_status=not_wired`**. ⚠️ **Today every live path reads raw config** — each `Live*Client` has an internal `if (! Config::get('integrations.X.live')) throw disabled` gate, factories select live/fake on a bare flag, and *non-binding logic* reads it too (`VoiceNoteProcessor` gates consent on `services.whisper.live`). **All of these must be migrated to `isLive(key)`** (the resolver), or the in-app go-live toggle has no effect and `not_wired` integrations stay reachable. No raw `Config::get('….live')` (or blank-key) decides a live path anywhere.
  - **Container bindings that pick live-vs-fake must consult the activation+wiring resolver — not raw config flags.** Several services choose their client at container-bind time from a bare flag: `AiServiceProvider` → `FakeAiClient` when `services.anthropic.key` is blank; `AppServiceProvider` → `LiveWhisperClient` when `services.whisper.live` is true; likewise payments. The rule: **`wiring_status=not_wired` ⇒ the binding is forced to the *fake* client regardless of the env flag** (so e.g. a stray `FEATURE_WHISPER_LIVE=true` can't select the auth-less, unwired `LiveWhisperClient`); **`wired` + `managed_via=vault` ⇒ the factory resolves the key via `IntegrationCredentials` and `live` via the activation resolver** (AI/payments), taking effect on the next request. No bare `config(...live)`/blank-key check decides a live binding on its own.
- **All reference-data writes are governed + append-only + source-attributed** (AI Integrity Principle). A manually entered figure carries who entered it, the source, and the as-at date — exactly like an API-sourced value. History is never overwritten; new values are new versioned snapshots. Routes through the existing Learning Update Queue (Layers 12/13).
- **No external API is required for *advisory* functionality.** Every integration must function (degrade) with no credential present. Turning an integration "live" is an explicit, reversible admin action — never the default. **Two honest exceptions with no manual equivalent:** **real payment collection** (Stripe/Windcave) and **real AI generation** (Anthropic). With those off, the platform still runs end-to-end but in defined degraded modes — **AI → `FakeAiClient`** (deterministic stub, clearly labelled "AI unavailable", never passed off as real analysis); **payments → sandbox/no-collection** (the sign-off flow completes but no money moves; the practice invoices outside the platform). These are labelled `api_required` in the registry, not "manual."
- **Audit + RLS everywhere.** New tables get RLS where client-scoped; credential + reference-data tables are platform-scoped, super-admin-gated. Every mutation audited via `AuditWriter`.

## 2. Stats NZ ADE — verified reality (drives WO-A05)

Live-tested this session with the real subscription key:
- **Works:** auth via `Ocp-Apim-Subscription-Key` header; 911 dataflows discoverable; real data retrieval confirmed (HTTP 200).
- **Format:** **SDMX-JSON v1.0** (`application/vnd.sdmx.data+json; version=1.0` for data, `…structure+json; version=1.0` for structure). `user-agent` header **required**. The `version` path segment must be **explicit** (e.g. `1.0`) — **`latest` is rejected**. Base URL: `https://apis.stats.govt.nz/ade-api` (our config currently has the wrong host).
- **Critical content gap:** ADE does **not** contain the headline macro series we stubbed — **no CPI, no GDP, no unemployment-rate** dataflows. It holds Census, **Business Demography** (enterprises by industry & employee-size), **LEED** (earnings), income, and population projections.
- **Therefore (path 3, owner-decided):** ADE supplies **industry/business benchmarking** data; **CPI/GDP/unemployment and anything else ADE lacks are entered manually** via WO-A03. Stats NZ becomes *both* an API integration (industry data) *and* a manual reference-data source (macro figures).

## 4. Work Orders

### WO-A01 — Admin Credentials Vault (encrypted)
- **Goal:** A super-admin-managed store of external-integration credentials, encrypted at rest, that integration clients resolve at point of use — replacing reliance on `.env` editing + redeploys for go-live.
- **Backend:**
  - `integration_credentials` table — `id`, `integration_key` (e.g. `stats_nz`, `google_calendar`, `stripe`), `field` (e.g. `api_key`, `client_id`, `client_secret`, `base_url`), **`value_envelope`** (`KeyEnvelope`-wrapped) + `value_envelope_meta`, `last_four` (for masked display), **`status` (`active` | `revoked`)** (set + rotate both leave it `active`; rotation is an audited *event*, not a state), `set_by_user_id`, `rotated_at`, `revoked_at`, timestamps. Unique `(integration_key, field)`. **Add `value_envelope` to `RewrapEnvelopes::sources()`** so PQC/key rotation covers it.
  - **Credential resolver** (`IntegrationCredentials` service): `get(integration_key, field)` → **if a DB row exists**: return the decrypted value when `active`, return **`null`/absent when `revoked` (terminal — no env fallback)**; **only when no DB row exists** → read the registry's declared **`config_path`/`env_fallback_path`** for that `(integration_key, field)` (not a guessed `integrations.{key}.{field}`, since paths are nested/varied). **For `managed_via=vault` credentials, every integration client + verifier reads its key/secret through this resolver**, never `Config::get`/`env()` directly (migrate `LiveStatsNzClient`, the AI/payment factories, and `PaymentWebhookVerifier`). **`managed_via=environment` (mail delivery) is exempt** — it intentionally reads Laravel's mail config/env (we are *not* rebinding `MailManager` to the vault).
  - **Permission + fresh step-up:** add `CREDENTIAL_MANAGE = 'credential.manage'` to `Permission` enum + `roleMatrix()` (super-admin only); reseed. Set/rotate/revoke routes gated on it **and a dedicated `require.fresh-step-up` middleware** that forces a fresh MFA challenge (verified within the last few minutes, else re-prompt) — *not* the passive `mfa` middleware, which only checks an already-verified session.
  - **Masking + redaction:** controller endpoints never return the plaintext or the envelope — only `last_four`, `status`, `set_by`, `rotated_at`. Set/rotate writes via `KeyEnvelope::encrypt`; audited (`credential.set`/`credential.rotated`/`credential.revoked`, value never logged). **Extend the `Redactor` with key-name-based scrubbing** (api_key/secret/token/client_secret/subscription_key/password/key) so a credential value can never leak into logs or audit context.
- **Frontend:** an admin "Integration Credentials" surface — list per integration with masked status + Set/Replace/Revoke actions (the value field is write-only; never pre-filled).
- **Tests:** value stored encrypted (raw column is ciphertext, not the key); endpoints never expose plaintext; **a newly-`set` and a `rotated` credential both resolve** (active); **a `revoked` row resolves as absent and does NOT fall back to env even when env holds a key** (terminal) + blocks go-live; resolver falls back to env **only when no DB row exists**; `credential.manage` gating (super-admin only; advisor 403); **fresh step-up enforced — an already-verified-but-stale session is re-challenged on set/rotate/revoke**; audit row written without the value; **key-name redaction in both `AuditWriter` context and `HealthRecorder` (an `integration_calls` row with a `?subscription-key=` endpoint or an auth-bearing `error_payload` is scrubbed)**; `RewrapEnvelopes` covers the new column.
- **Acceptance:** a super-admin (after a fresh step-up) can set/rotate/revoke an integration key in-app; it's encrypted at rest, never displayed or logged; the client resolves only active credentials; revoked keys stop resolving immediately; non-super-admins cannot.

### WO-A02 — Go-Live / Integrations Readiness panel
- **Goal:** Turn the existing Integration Health surface into the "API navigation panel" the owner wants — a single place to see what each integration needs and to go live.
- **Depends on:** WO-A01, `IntegrationHealthController`.

### WO-A03 — Governed Reference Data Management (manual entry + upload)
- **Goal:** A super-admin capability to enter or upload the reference figures the platform needs but no API provides — economic indicators (CPI, GDP, unemployment, OCR if not from RBNZ), valuation multiples, industry WACC, cost-per-beneficiary benchmarks — governed, append-only, source-attributed.
- **Depends on:** the existing `economic_indicators` / valuation-multiple / `industry_wacc_data` / benchmark tables + the Learning Update Queue (Layers 12/13).
- **Backend:**
  - ⚠️ **Append-only ledger (truly immutable) + governance state on the linked `LearningUpdate`.** Consumer shapes differ per dataset — `economic_indicators` is scalar `unique(indicator,period_date,source)`+`updateOrCreate`; **valuation_multiples** is `industry_code/metric/multiple_low/mid/high/quarter` (+ `superseded_at`); **industry_wacc_data** is `wacc_rate`+component rates; **cost-per-beneficiary** reads the **latest *implemented* Layer-36 `LearningUpdate`** (`proposed_change.benchmarks`), *not a table* — ⚠️ today it reads `[APPROVED, IMPLEMENTED]`; this WO tightens it to `IMPLEMENTED`-only (see the projector bullet). So add a dedicated **`reference_data_entries` ledger** that is **immutable** — `id`, `dataset` (`economic_indicator`/`valuation_multiple`/`industry_wacc`/`cpb_benchmark`), **`payload` (jsonb — the full dataset-shaped record)**, `as_at`, `source`, `entered_by_user_id`, **`learning_update_id`** (the governance candidate this submission created), `created_at`. **No mutable status on the ledger** — pending/approved/rejected lives on the linked `LearningUpdate` (the queue's existing machinery), so the ledger never gets updated after insert.
  - **Per-dataset projectors (heterogeneous), run only on the queue's *implement* step:** `economic_indicator` → `economic_indicators` (`updateOrCreate`, `source_badge='manual_admin'`); `valuation_multiple` → `valuation_multiples` (its `superseded_at` versioning); `industry_wacc` → `industry_wacc_data`; **`cpb_benchmark` → there is no table — the submission *is* a Layer-36 `LearningUpdate` whose `proposed_change.benchmarks` the `NpoValueCalculator` reads. ⚠️ Today the calculator reads status `[APPROVED, IMPLEMENTED]`, so an approved-not-implemented update is already live — *tighten the CPB read to `IMPLEMENTED` only*** so it matches the "effect on implement" guarantee uniformly with the table-projecting datasets.
  - **Governance gates effect (no early influence).** Submitting an entry inserts the immutable ledger row **and** creates a `pending` `LearningUpdate` (Layer 12/13/36 as appropriate); it has **zero effect** until the queue **approves+implements**. Table-projecting datasets project on implement; CPB takes effect only when the calculator reads the now-latest-**implemented** update (after the tightening). Rejected → never projects/takes effect. Audited throughout.
  - **CSV/Excel upload** option: parse a **per-dataset template** (the columns that dataset needs) into a `pending` ledger entry; file goes through `SecureFileWriter` (scan-before-store).
  - **Permission:** reuse/extend an admin reference-data permission (super-admin); gated + audited.
- **Frontend:** an admin "Reference Data" surface — per dataset: current value(s) + as-at + source, a "Record new value" form, and a "Upload CSV" option. Shows history (append-only).
- **Tests:** for **each dataset** the projector targets the right consumer correctly — economic→`economic_indicators`, valuation→`valuation_multiples` (superseded versioning), WACC→`industry_wacc_data`, **CPB→a Layer-36 `LearningUpdate` the `NpoValueCalculator` then reads** (not a table); **a `pending` entry (its `LearningUpdate` unapproved) does NOT affect any calculation** — the cardinal governance test; **for CPB specifically, an `APPROVED`-but-not-`IMPLEMENTED` update also does NOT yet affect the calculation** (proves the implemented-only tightening); rejected never takes effect; the **ledger row is never mutated** after insert (governance state is on the linked `LearningUpdate`); source + as-at attribution present; CSV upload parses the per-dataset template + scans; super-admin gating; no calculation reads a value lacking attribution.
- **Acceptance:** a super-admin can keep economic indicators / valuation multiples / WACC / benchmarks current by hand or by per-dataset upload — each fully attributed, append-only, and **only affecting calculations after the governed approve+implement step** (merely-approved/pending never does), with no API.

### WO-A04 — Dashboard reference-data tasks (cadence-aware)
- **Goal:** Make keeping reference data current a **routine, prompted task**, not a memory test — a recurring item on the admin dashboard.
- **Depends on:** WO-A03, the admin dashboard + notification infra.
- **Backend:** each reference dataset declares a **refresh cadence** (OCR ~8×/yr, CPI/multiples quarterly, WACC/benchmarks per their cycle). A scheduled check marks a dataset **stale** by the **latest *implemented*/effective** value's `as_at` (`now - implemented_as_at > cadence`) — **not** the latest approved/pending, so the dashboard never shows "fresh" before the calculation actually uses the value. Generates an admin task + notification ("CPI quarterly update due — last updated [date]"); the task clears only when a new value is **implemented**.
- **Frontend:** a "Reference Data Updates" panel on the admin dashboard listing due/overdue datasets with a direct link to the WO-A03 entry form.
- **Tests:** a stale dataset raises a task; **only an *implemented* new value clears it — a merely-pending or merely-approved value does NOT** (freshness tracks implemented data, matching when the calculation actually uses it); overdue escalates; only super-admin sees it.
- **Acceptance:** the admin is reliably prompted when any reference figure is due, and one click takes them to record it.

## 5. Open decisions / assumptions (recommended defaults)
1. **Credential storage** — default: `KeyEnvelope`-encrypted DB vault, resolver precedence: active DB row → env only if no DB row → **revoked is terminal** (no env fallback). (Not a plaintext settings table — rejected on security grounds.)
2. **Credential scope + exact fields + paths** — per `(integration_key, field)`, each declaring an explicit **`config_path` + `env_fallback_path`** (paths are nested/varied: `integrations.calendar.*`, `integrations.accounting.*`, `services.anthropic.key`, …) and the **exact** field set per integration (Stripe `secret`+`webhook_secret`; Windcave `api_user`+`api_key`+`webhook_secret`) so go-live can't pass with a missing webhook secret and the resolver never guesses a path.
3. **Reference-data governance** — default: an **immutable `reference_data_entries` ledger** (`dataset` + jsonb `payload` + `learning_update_id`) is the submission record; **governance state (pending/approved/rejected) lives on the linked `LearningUpdate`** (Layers 12/13/36), not the ledger. **Per-dataset projectors run on the queue's *implement* step**: economic→`economic_indicators`, valuation→`valuation_multiples`, WACC→`industry_wacc_data`, **CPB→a Layer-36 `LearningUpdate`** the `NpoValueCalculator` reads **(tightened to `IMPLEMENTED`-only)**. Nothing affects calculations until implemented.
4. **Activation state** — default: a DB-backed **`integration_activations`** table + resolver (`isLive` = DB activation else config flag, **`false` if `not_wired`**); the toggle writes it. **Every live check uses `isLive`** — each `Live*Client`'s internal `.live` gate, the AI/payment/Whisper factory selection, **and** non-binding logic gates (`VoiceNoteProcessor`) — never raw `Config::get('….live')`. This client-by-client migration is the substantive backend of A02.
5. **Step-up for sensitive admin actions** — default: a dedicated **fresh-step-up middleware** (re-challenge MFA now) on credential set/rotate/revoke + go-live — not the passive `mfa` session check.
6. **`wiring_status`** — default: registry rows declare `wired`/`not_wired`; the current **not_wired** set is **PPSR, LINZ, IPONZ, NZ Parliament, WorkSafe, Whisper** (no Live client / no key path), which disables Test + Go-live until their live client + config + resolver wiring is added. (Incorporated Societies is wired via the Companies Office live client.)
6a. **`managed_via`** — default: most integrations are `vault` (admin-entered + go-live toggle). **Email/mail delivery is `environment`** — configured via `MAIL_*`/transport at deploy, shown on the panel as **transport-specific** read-only readiness only (no in-app entry/activation; rebinding `MailManager` to a DB vault is out of scope). **`SesSendGrid` is `managed_via=environment`, `wiring_status=not_wired`, not vault-activatable** (a fake-only, non-production mail scaffold) — hence intentionally absent from the *vault* not_wired disable-list. (In-app mail management is a possible future feature.)
7. **Calendar primacy** — default: built-in calendar is primary + always-on; Google/Microsoft sync is optional, off by default, creds via the vault.
8. **Stats NZ split** — default: ADE for industry/business benchmarks; CPI/GDP/unemployment via manual reference data (+ RBNZ where it publishes them).
9. **Macro indicators home** — default: `indicators()` reads the latest manual snapshot; if RBNZ live is later enabled for OCR, it takes precedence for that one indicator.

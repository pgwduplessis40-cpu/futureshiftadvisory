> **v1.9 revision (legacy offline records).** One closure: (ff) **Records already in users' IndexedDB at deploy have no `client_id`.** The shipped `QueueRecord` type (`portal-offline.ts` line 8) and `queueRecord()` (line ~223) persist only `kind/url/dedupeKey/payload`, so pre-deploy records can't supply `X-Portal-Client-Id`. The flush path must **branch on a missing `client_id`: keep + surface for manual re-submit, and never fall back to the server's `resolveFor()`** (which would re-open the wrong-client bug). Adding the field is backward-compatible (optional property, no IndexedDB version bump); a **legacy-record test** asserts a no-`client_id` record is never replayed via `resolveFor()`.
>
> **v1.8 revision (WO-122 implementation-ready).** Three final WO-122 closures: (cc) **Capture the client at queue time.** The queue stores no client and the URLs are generic, while `ClientPortalResolver` resolves `latest()` accessible client — so offline work could replay into the **wrong client**. Capture `client_id` at enqueue (`X-Portal-Client-Id`, folded into the dedupe key), key the ledger on the **queued** client, and **validate it's still accessible on replay (else 403, keep)** — never trust the replay-time `resolveFor()`. (dd) **All five redirect sources, not one.** Beyond `auth`/CSRF/`RequireMfa`, the manual-redirect middleware `EnforceSessionSecurity` (session-expiry/step-up), `RequireAcceptedTerms`, and `EnsureEmailIsVerified` also bounce sync requests — so a **single outer "sync response normalizer"** rewrites any auth-flow redirect → JSON 401 (plus the exception layer for `auth`/CSRF), with a test per source. (ee) **Server-side request fingerprint.** The client document dedupe key uses filename/size/lastModified (not content), so a different file can collide; the server stores a content/body `request_fingerprint` — same key + different fingerprint → **409 Conflict** (kept), not a stale cached success.
>
> **v1.7 revision (request/middleware/data shapes).** Three tightenings against the real auth stack and controllers: (z) **Auth/MFA redirect-to-JSON can't be in-group middleware** — portal routes sit behind `['auth','verified','mfa']` (`routes/portal.php` line 14) which redirect *before* the controller, and `RequireMfa` returns a **manual redirect** (not an exception), so the sync 401/419 handling lives in the **exception layer** (`shouldRenderJsonWhen` keyed on `X-Portal-Offline-Sync`, **not** `expectsJson()` — the queue sends `Accept: text/html`) **plus** a header guard in `RequireMfa`. (aa) **Replay needs a normalized JSON success to cache** — `OnboardingController::store` returns a `RedirectResponse`, so for sync requests the submit handlers return normalized JSON (`{ok,operation,resource_id?}`) and *that* is what the ledger stores/replays. (bb) **Draft provenance fixed** — there is no generic "engagement" row (the default trigger `OffboardingService` creates an `OffboardingRecord`), so `knowledge_entry_drafts` uses **polymorphic `source_type`/`source_id`** (or concrete nullable FKs) + `source_reference`, not `source_engagement_id`.
>
> **v1.6 revision (hand-off traps).** Three implementation traps cleared before hand-off: (w) **Residual `before_state={status:'draft'}`-only mentions aligned** — the WO-121 tests line and §4 decision #1 now also require `learning_update_implementation_id: null` in `before_state` and assert rollback clears the FK (matching the v1.5 layer/table text). (x) **WO-122 replay target made explicit** — the client posts **directly to `record.url`** (the existing questionnaire-submit + document-upload routes), so the hardening lands as a **sync-aware middleware on those existing routes** (triggered by the `X-Portal-Offline-Sync` header), *or* a dedicated endpoint **with** a matching `record.url` change — never a new endpoint the PWA doesn't call. (y) **WO-123 trigger doesn't exist** — there is **no `app/Events/` / `EngagementCompleted`** in the app; the WO must create the trigger (default: a direct call from `OffboardingService`) or build+dispatch a new event from the concrete completion path(s), with a test that real completion emits a draft.

> **v1.5 revision (consistency sweep).** Three corrections: (t) **Stale risk row fixed** — risk **GC-3** still said `clearPortalOfflineQueue()` runs on "logout + session-expiry", contradicting the v1.4 decision (clear on **explicit logout only**; session-expiry keeps + retries). Aligned. ## 1. Non-negotiable principles (carried forward — still binding)

Everything in `CLAUDE.md` and the phase plans remains in force. The ones these WOs stress hardest:

- **No silent learning (spec §3, §23) — two *distinct* governance models, do not conflate them.**
  - **WO-121 Template Library → admin learning-queue model.** Templates are practice-wide, so the AI **emits a governed `learning_updates` candidate** (`automatic_application=false`, source-attributed); **approval ≠ creation** — `ApprovalFlow`/`assertImplementationAllowed()` only *authorises*; a **separate `TemplateImplementer`** then writes a `LearningUpdateImplementation` ledger row and **activates** the template (WO-93 implementer pattern). No template is *active/usable* on detection or on approval alone.
  - **WO-123 Knowledge capture → advisor-draft model (NOT the admin learning queue).** A knowledge entry is **personal to one advisor** (the existing `knowledge_entries` table is per-advisor RLS), so it does **not** go through admin approval. Instead a **newly-wired completion trigger** (a direct call from `OffboardingService`, or a created-and-dispatched `EngagementCompleted` event — there is no event bus today; see WO-123) or a manual advisor action writes a **per-advisor draft**; the **advisor** reviews and accepts/edits/discards → accept creates *their own* live `knowledge_entries` row. **No `learning_updates` candidate, no `ApprovalFlow`, no `LearningUpdateImplementation`.** The shared invariant both models honour: **nothing is auto-published / auto-activated** — a human (admin for templates, the owning advisor for knowledge) must act.
  - **Layer IDs (do not reuse 16/17) — only ONE new registry layer.** `LayerCadenceRegistry::definitions()` already occupies **all of 1–32** (16 = Questionnaire optimisation, 17 = Coach referral calibration; `QuestionnaireOptimisationLayer::LAYER_ID = 16`). Because **knowledge capture is event-triggered + per-advisor, it is NOT a cadence-registry layer** — only the **Template Suggestion layer** is registered, as **ID 33** (`TemplateSuggestionLayer`). Do **not** renumber 16/17. The spec's conceptual "layer 16 knowledge / layer 17 template" labels are honoured by feature, not by registry index. Update `docs/v2.4-spec-coverage.md` §23 to record the registry now carries **33** cadenced layers (32 + template), with knowledge capture as an advisor-draft AI-assist.
  - **The hard-coded "32" assertions must be updated to 33 (they fail otherwise).** Adding the template layer changes the registered-layer count: `tests/Feature/Admin/LearningCadenceTest.php` (the `test_registry_contains_thirty_two_layers...` method — `assertCount(32, …)`, `assertCount(32, $runs)`, `assertSame(32, LearningLayerRun::count())`) and `tests/Feature/Learning/ActiveLayerEngineTest.php` (five `32` assertions: states, runs, active states, `candidates_created`, detected `LearningUpdate`s), plus any `LearningMonitorDashboard` count assertion. **WO-121 updates them all to 33** (and renames the `thirty_two` test). **WO-123 does not touch them** (it adds no registry layer), so there is **no order dependence**. Grep `thirty_two` and `\b32\b` across `tests/Feature/{Admin,Learning}` in WO-121.
- **AI Integrity (spec §3).** AI-drafted content (template suggestions, knowledge capture) is **evidence-based and source-attributed** via `AiClient`; `FakeAiClient` in tests. No fabricated content; honest where evidence is thin.
- **All external calls through `ResilientHttp` + feature flags (spec §4.2).** The **Calendar (WO-124)** and **accounting (WO-125)** integrations follow the WO-05/WO-37 pattern: feature-flagged (default OFF → fixture/fake), retry/breaker/cached-fallback, tokens in `KeyEnvelope`, every call audited. No raw `Http::` to a third-party host.
- **Security baseline (spec §4).** **Offline/PWA (WO-122)** *hardens the already-shipped WO-101 portal PWA* (encrypted IndexedDB queue, service worker, manifest). The hardening focus is **server-side**: synced documents re-run the **virus scan** (`SecureFileWriter` scan-before-store) and **RLS**; the sync endpoint re-authenticates and is **server-side idempotent** (replay-safe, dedupe-key enforced in the DB, not just the client). No PII in logs (Redactor). MFA/invite-only unaffected.
- **Audit + RLS everywhere.** New client-scoped tables get `ENABLE`+`FORCE` RLS + `*_client_scope` policy; every mutation audited via `AuditWriter`.

## 3. Work Orders

### WO-121 — Template Library (governed AI-suggested templates)
- **Goal:** A practice template library where the AI **suggests** new templates from completed engagements and **admin approves** via the governed learning queue (§11; §23 layer 17). Nothing is auto-*activated* — suggestions land as dormant `draft` rows and only become usable after approval + the implementer.
- **Depends on:** the learning queue (`learning_updates`, `ApprovalFlow`) + `AiClient`.
- **Backend:**
  - `templates` table — `id`, `category` (report/proposal/email/plan-section/…), `title`, `body`/`structure` (jsonb), `source_reference` (the engagement/report it was distilled from), `status` (`draft`/`active`/`archived`), `version`, `created_by_user_id`, **`learning_update_implementation_id` (nullable** — the dormant `draft` is created at detection *before* any ledger row exists; the implementer sets it on activation), timestamps. **Practice-wide, internal** (not client-scoped) → no client RLS; the table holds no client PII (templates are generalised). **`draft` is the dormant candidate state** (created at detection, excluded from every usable/active surface); `active` is the only usable state; the implementer flips `draft`→`active` and rollback flips it back (see the layer bullet).
  - **Permission model (enum + role matrix, NOT migration rows).** This app's permissions are the `App\Enums\Permission` enum, assigned via `Permission::roleMatrix()` and seeded by `RoleSeeder`→`valuesForRole()` (consuming `PermissionSeeder`). So add **two enum cases** — `TEMPLATE_MANAGE = 'template.manage'` and `TEMPLATE_VIEW = 'template.view'` (do **not** overload `knowledge.manage`) — then wire them into `roleMatrix()`: super-admin already gets all via `self::cases()`; give **advisors `TEMPLATE_VIEW` only** (and any admin role `TEMPLATE_MANAGE`). Re-run the seeder so roles actually receive them. Gate controller **write** actions on `template.manage` (mutation super-admin/admin-only) and **read** actions on `template.view` (advisors view/use). A migration is **not** how the permission is created — only the `templates` table is a migration.
  - A **new layer-33 learning layer** (`TemplateSuggestionLayer`, `LAYER_ID = 33` — the only new registry layer; see §1; **not** 17) registered in `LayerCadenceRegistry` that, after engagements complete, distils candidate templates via `AiClient`. On detection it persists the `templates` row as a **dormant `draft`** (never listed/usable) **and** writes a **`learning_updates` candidate** (`automatic_application=false`, source-attributed) referencing it. **Approval does not activate the template.** On approval, `ApprovalFlow`/`assertImplementationAllowed()` only authorises; a **separate `TemplateImplementer` job/step** then **flips the row `draft`→`active`**, sets `learning_update_implementation_id`, and writes the `LearningUpdateImplementation` with **`target_type`/`target_id` = the template, `before_state = {status:'draft', learning_update_implementation_id: null}`, `after_state = {status:'active', learning_update_implementation_id: <id>}`**. Capturing the FK as `null` in `before_state` matters: `Rollback::restorePriorState` `forceFill`s **only the fields present in `before_state`** (`Rollback.php` line 104) and no-ops on empty (line 89), so listing both fields reverts `active`→`draft` **and** clears the implementation link — **no row deletion, no edit to the service every layer shares**. Never activate on detection or on approval alone.
- **Frontend:** an internal "Templates" surface — **read for advisors, manage for admin/super-admin** (`template.manage`); list/active/archived + view; suggested-template candidates appear in the existing Learning Update queue UI for approve/reject.
- **Tests:** the layer emits a candidate + a **dormant `draft`** row (no *active* template, draft not in any usable list); **approval alone does not activate** (the implementer must run); the implementer flips `draft`→`active` **and** writes a `LearningUpdateImplementation` with **`before_state={status:'draft', learning_update_implementation_id: null}`**; **rollback reverts `active`→`draft` AND clears `learning_update_implementation_id` back to `null`** via the shared `Rollback` (asserts the template is dormant *and* unlinked again, not deleted); reject leaves it `draft`; **`template.manage` gating** (advisor 403 on mutate, 200 on view); `FakeAiClient`; source attribution present; no PII in the template body/logs; **registry now has 33 layers** (update the `LearningCadenceTest`/`ActiveLayerEngineTest`/dashboard assertions in this WO — see §1).
- **Acceptance:** AI suggests templates that become *active/usable* **only after approval *and* the implementer flips `draft`→`active`**; rollback returns them to dormant `draft`; advisors can view/use but not mutate; library is internal-only.
- **Files:** migration (`templates` table only) + `Template` model; `App\Enums\Permission` (+`TEMPLATE_MANAGE`/`TEMPLATE_VIEW` cases + `roleMatrix()` assignments) + reseed via `RoleSeeder`/`PermissionSeeder`; `TemplateSuggestionLayer` registered in `LayerCadenceRegistry` (ID 33); `TemplateImplementer` (approval→activation `draft`→`active`, writes ledger with reversible `before_state`); `TemplateController` (read vs `template.manage` write) + routes; admin/advisor template page; **the `32`→`33` test updates** (`LearningCadenceTest`, `ActiveLayerEngineTest`, dashboard).

### WO-122 — Offline mode / PWA — **harden the existing WO-101 layer**
- **Already built (WO-101 — do not rebuild):** the portal PWA exists today — `public/manifest.webmanifest` (name "Future Shift Advisory Portal", `start_url=/portal`), `public/sw.js` (portal fallback, `'portal-offline-sync'` / `PORTAL_OFFLINE_SYNC` background sync), `public/offline.html`, and `resources/js/lib/portal-offline.ts` (encrypted IndexedDB queue via `crypto.subtle`, `dedupeKey` index, `queueQuestionnaireSubmission`, `queueDocumentUpload`, `flushPortalOfflineQueue`). `tests/Feature/Portal/OfflinePwaTest.php` proves the client-side offline flow (encrypted at rest, dedupe keys, background sync, offline forms). **The client/browser surface is done.**
- **Goal (this WO):** close the **server-side** gaps the client queue assumes — make the sync endpoint replay-safe, scanned, scoped, and re-authenticated — plus stronger browser QA. This is hardening, not a new build.
- **Depends on:** WO-101 (the PWA layer), WO-18 (`SecureFileWriter` upload), WO-16/17 (portal/questionnaire).
- **Frontend transport change (required first — the client sends neither the key nor the client today).** `portal-offline.ts::sendRecord` currently sends **only** `X-Portal-Offline-Sync: '1'`; the `dedupeKey` is computed/stored in IndexedDB but **never transmitted**, and the queued record does **not** capture which client the work was for (`queueRecord` stores `kind/url/dedupeKey/payload` only — `portal-offline.ts` line ~88/109). Two additions, both required:
  - **`X-Idempotency-Key: record.dedupeKey`** on both the questionnaire (JSON) and document-upload (FormData) `fetch` calls.
  - **Capture the resolved `client_id` at queue time** (from the portal page props / a `<meta name="portal-client-id">`), store it **on the queue record and fold it into the `dedupeKey`**, and send it on replay as **`X-Portal-Client-Id`**. This is essential because `ClientPortalResolver::resolveFor` picks the **`latest()` accessible client** (`ClientPortalResolver.php` line 25) and the queued URLs are generic (`/portal/onboarding/{step}` — `OnboardingController.php` line ~129); without the captured client, offline work could **replay into the wrong client** if access changes or the user has multiple clients.
  - **Legacy records already in IndexedDB have no `client_id` — handle them explicitly.** The shipped `QueueRecord` type (`portal-offline.ts` line 8) is `{id, kind, url, dedupeKey, createdAt, attempts, payload}` and `queueRecord()` (line ~223) persists only those, so any record enqueued **before this deploy** lacks `client_id`. On flush, a record with **no `client_id` must NOT be replayed** (and must **never** fall back to the server's `resolveFor()` — that re-opens the wrong-client bug). Instead **keep it and surface it for manual re-submit** (a one-time "saved before an update — please re-submit" prompt that re-queues it *with* a `client_id`, or lets the user discard). Adding the field is backward-compatible (a new optional property; no IndexedDB version bump / index needed), but the **flush path must branch on its absence**.
  Keep `OfflinePwaTest` green and assert both headers are sent.
- **Backend (the actual work):**
  - **Where the server work lands — harden the routes the queue *already* posts to (default).** Today the client does **not** call a dedicated sync endpoint: `portal-offline.ts::sendRecord` replays **directly to `record.url`** — the original questionnaire/onboarding-submit route (line ~326) and document-upload route (line ~359) — with the `X-Portal-Offline-Sync: '1'` header. So the hardening attaches to **those existing target routes**, split by *where each concern can run*: **(idempotency + scan)** run in the controller path (a sync-aware middleware inside the group, or in-controller, gated on the header) — these only execute once auth/MFA have *passed*, which is correct; **(redirect→JSON 401/419)** can **NOT** be in-group middleware because `auth`/`verified`/`mfa` redirect first — it lives in the exception layer + a `RequireMfa` header guard (see the re-auth bullet). **Alternative (only if cleaner separation is wanted):** add a dedicated replay endpoint **and** change the queue's `record.url`/`sendRecord` to post there — never add the endpoint without also repointing the client; even then the auth/MFA redirect→JSON handling is unchanged (it's upstream of any route). Pick one explicitly; do not leave server work in a route the PWA doesn't hit.
  - **Server-side idempotency (validate the *queued* client; fingerprint the request).** The replay target reads `X-Idempotency-Key` (the client `dedupeKey`) and **`X-Portal-Client-Id`** (the client captured at queue time) and:
    - **Validates the queued client against access at replay time** — the `X-Portal-Client-Id` must be in the user's current `accessibleClientIds()` (and not suspended); if it isn't (access revoked, etc.), **reject (403) and keep the record**, never silently retarget. The dedupe ledger keys on **`(user_id, client_id, operation/route, idempotency_key)`** using the **queued `client_id`** — **not** the replay-time `resolveFor()` result (which is `latest()` and can drift), and **not** `(user, key)` alone.
    - **Stores a server-computed request fingerprint** alongside the key — a hash of the **actual payload** (for uploads: the **file *content***, server-side; for questionnaire/onboarding: the normalized body), because the client `dedupeKey` for documents is computed from **filename/size/lastModified/fields, NOT content** (`portal-offline.ts` line ~109), so two different files can collide on the key. On replay: **same key + same fingerprint → return the cached prior result**; **same key + different fingerprint → `409 Conflict`** (the client keeps it / surfaces it, never treats it as synced). 
    - **Stores the prior response payload (or created-resource ref)** so a matching replay **returns that stored result deterministically** rather than re-executing — never double-creating a submission or document. Idempotency must not rely on the client alone.
    - **Normalize the cached response (questionnaire/onboarding return redirects today).** `OnboardingController::store` returns a `RedirectResponse` (`to_route('portal.dashboard')` on the review step — `OnboardingController.php` line ~90), and document upload already returns JSON. A redirect is not a meaningful cached "prior result", so for **sync requests** the submit handlers must return a **normalized JSON success** (e.g. `{ok:true, operation, resource_id?, status_slug}`) — keyed on the `X-Portal-Offline-Sync` header, same as the error path — and **that** JSON is what the ledger stores and what a replay returns. This keeps ledger semantics uniform across uploads *and* questionnaire/onboarding submissions.
  - **Scanned upload replay:** queued document uploads replay through the existing upload path so every file goes through **`SecureFileWriter` (virus-scan-before-store)** on sync — no offline-queued file bypasses scanning.
  - **Re-auth / session handling — "redirect-to-login ≠ success" (sharp contract).** Today `flushPortalOfflineQueue` deletes a queued record on **any** `response.ok` (`portal-offline.ts` line ~160), and the questionnaire replay uses `redirect: 'follow'` with an HTML `Accept` header (line ~326). So when the session has expired, the auth middleware **302-redirects to `/login`**, `fetch` follows it, and the login page returns **200 HTML** — the client treats the record as **synced and deletes it**, silently losing the submission. The fix has two halves. **(server) — one normalizer + the exception layer cover ALL redirect sources (there are five, not one).** A sync request can be bounced by the `auth` guard (throws `AuthenticationException`), CSRF (`TokenMismatchException` → 419), **and four middleware that return *manual* redirects**: `RequireMfa` (`RequireMfa.php` 33/37 → `mfa.setup`/`mfa.challenge`), `EnforceSessionSecurity` (`EnforceSessionSecurity.php` 43/59 → `login`/step-up `mfa.challenge`), `RequireAcceptedTerms` (`RequireAcceptedTerms.php` 36/40 → `terms.declined`/`terms.pending`), and `EnsureEmailIsVerified` (the `verified` middleware → `verification.notice`). Because several run **before** the controller and **return** (don't throw), patching one is not enough. Use both: **(i) exception layer** — `shouldRenderJsonWhen(fn($req) => $req->hasHeader('X-Portal-Offline-Sync'))` so `AuthenticationException`→401 and `TokenMismatchException`→419 render as JSON (**keyed on the header, NOT `expectsJson()`** — the queue's questionnaire `fetch` sends `Accept: text/html`, `portal-offline.ts` line ~331); **(ii) a single outer "sync response normalizer" middleware**, registered **ahead of** the web/portal auth stack, that post-processes the response: if the request carries `X-Portal-Offline-Sync` **and** the response is a **3xx redirect to an auth-flow route** (`login`, `mfa.*`, step-up, `terms.*`, `verification.*`), rewrite it to **JSON 401** — this catches all four manual-redirect middleware in one place rather than editing each. (Editing each middleware individually is the acceptable fallback, but then **all four** must be done + tested.) **(client)** `sendRecord` must treat a redirect-to-login (or 401/419, or a response whose final URL/content is an auth-flow page) as **failure → keep the record + increment attempts**, never as success. Sync respects **RLS** (a client only writes their own scope).
  - **Clear-on-explicit-logout only (NOT on session-expiry) — these are opposite outcomes, pick one.** The AES-GCM key is in `localStorage['fsa.portal.offline.key.v1']` and the queue in IndexedDB `fsa-portal-offline-v1`; `UserMenuContent.handleLogout` only calls `router.flushAll()`. Add a `clearPortalOfflineQueue()` to `portal-offline.ts` (deletes the IndexedDB database + removes the localStorage key) and invoke it **only from `handleLogout`** (explicit user logout). **Do not** clear on a 401/session-expiry from the sync path — that path is the *keep + retry* contract above (deleting there would destroy the very data the expired-session rule preserves). So: **session expired → keep the queue and re-prompt for auth; explicit logout → wipe the queue + key.**
- **Frontend:** verify/finish the offline + syncing indicator wiring to the WO-101 queue (it already exists — confirm states surface); wire `clearPortalOfflineQueue()` into logout. No framework swap.
- **Tests:** feature tests that a queued questionnaire submission and a queued upload **replay idempotently server-side** (dedupe via `X-Idempotency-Key` + `X-Portal-Client-Id`, no duplicates on double-sync) and a replay **returns the stored prior response** (deterministic, no re-execution); a **wrong/changed-client test** — a record queued for client A replayed when access has changed (or under client B) is **rejected (403) and kept**, never retargeted; a **legacy-record test** — a record with **no `client_id`** (queued before this deploy) is **never replayed via `resolveFor()`**; it is kept and surfaced for re-submit; a **same-key-different-content test** — same `dedupeKey`, different file content → **`409 Conflict`** (server fingerprint differs), record kept, *not* a false cached-success; a **same-key-different-client test** (two accessible clients → no false collision because the ledger key uses the **queued** `client_id`); the upload is **scanned on sync** (`SecureFileWriter`), RLS holds; a **normalized-JSON-success** test (a sync questionnaire/onboarding submit returns JSON, not a redirect, and that JSON is what a replay returns); **auth-flow-redirect → JSON tests keyed on the header (Accept: text/html), one per source** — unauthenticated→**401**, CSRF→**419**, MFA-not-verified→**401**, **session-expiry/step-up→401**, **terms-pending/declined→401**, **email-unverified→401** (none leaks a redirect/200), and the client **keeps the record** in each; a **clear-on-logout** test (queue + key gone) and a **session-expiry-does-NOT-clear** test; keep the existing `OfflinePwaTest` green. *(Browser-true offline behaviour stays manual-QA; the sync endpoints/contracts are the automated surface.)*
- **Acceptance:** the WO-101 offline queue syncs **exactly once** server-side (DB-enforced via the transmitted key + fingerprint), each synced item lands in the **client it was queued for** (or is rejected, never misrouted), every synced file is scanned and RLS-scoped, every auth-flow bounce degrades to JSON-keep-and-retry, and **logout clears the local queue + key** — with no regression to the shipped PWA.
- **Files:** `portal-offline.ts` (`X-Idempotency-Key` + `X-Portal-Client-Id` in `sendRecord`; add optional `client_id` to the `QueueRecord` type + capture at queue time + fold into `dedupeKey`; **flush branch for legacy no-`client_id` records → keep + surface, never `resolveFor()`**; `clearPortalOfflineQueue()` from logout only); `user-menu-content.tsx` (call clear on logout); a **sync-aware middleware** on the **existing questionnaire/onboarding-submit + document-upload routes** (idempotency + queued-client validation + server fingerprint + `SecureFileWriter` replay + normalized-JSON success), triggered by `X-Portal-Offline-Sync` — *or* a dedicated replay endpoint **plus** the matching `record.url` change (pick one); **a single outer "sync response normalizer" middleware** (ahead of the auth stack) that rewrites auth-flow redirects → JSON 401 + **`bootstrap/app.php` `shouldRenderJsonWhen`** for the `auth`/CSRF exceptions; `OnboardingController`/questionnaire submit (normalized JSON success for sync); a migration for the dedupe ledger (`user_id`, **`client_id`** (queued), `operation/route`, `idempotency_key` unique + `request_fingerprint` + stored `response` payload); new server-side + clear-on-logout tests. **No** `vite-plugin-pwa` swap, **no** new manifest/service worker — WO-101 owns those.

## 4. Open decisions / assumptions (recommended defaults)
1. **Template scope + permission + rollback (WO-121)** — default: **practice-wide internal** templates (no client PII); a **distinct `template.manage` permission**, mutation **super-admin/admin-only**, advisors **view/use**. **Activation** runs through a **separate `TemplateImplementer`** after approval (writes a `LearningUpdateImplementation` ledger row) — approval alone never activates. **Rollback = status-flip**: the row is created dormant (`draft`) with a **nullable** `learning_update_implementation_id`, the implementer flips `draft`→`active` with `before_state={status:'draft', learning_update_implementation_id: null}`, so the shared `Rollback` reverts `active`→`draft` **and** clears the FK (no deletion, no change to the shared service). If templates should be client-scoped, add RLS instead.
   - *Alternative (not preferred):* implementer creates an `active` row and rollback is taught to delete/archive create-type targets — but that **edits the shared `Rollback` service every layer uses**, so only do it if a dormant `draft` row is unacceptable.
2. **WO-122 is hardening, not a build** — WO-101 already shipped the PWA (manifest, `sw.js`, `offline.html`, encrypted IndexedDB queue). **Do not** introduce `vite-plugin-pwa` or a second service worker. The work is server-side: **DB-enforced idempotency, scanned upload replay, re-auth/RLS on sync.**
3. **Knowledge draft storage (WO-123)** — default: **Option A — a sibling `knowledge_entry_drafts` table** (drafts physically cannot leak into the live `knowledge_entries` index). Option B (draft columns on `knowledge_entries` + read-path filter) only if a single table is preferred and every read path is updated.
4. **Knowledge capture trigger (WO-123) — must be created (no event exists today).** Default: a **direct call from `OffboardingService`** (the concrete engagement-completion path) **plus** a manual advisor "draft from this engagement" action. Alternative: introduce + dispatch a new `EngagementCompleted` event from every completion path. Either way, wire it to real completion code and test that completion emits a draft — do **not** assume an event bus (there is no `app/Events/`).
5. **Calendar conflict rule + external-event storage (WO-124)** — default: FSA `meetings` are the source of truth for FSA-created events; external events are read-only in the meetings view; dedupe by **`calendar_event_mappings.(connection, external_event_id)`**, incremental via the stored `sync_token`/`delta_link`. Because `meetings.client_id` is NOT NULL, **external-only events store their payload on `calendar_event_mappings`** (title/start/end/location/attendees + `is_external_only`), not as `meetings` rows.
6. **Learning-layer registration (WO-121 only)** — default: **append ONE layer, ID 33 (`TemplateSuggestionLayer`)** to `LayerCadenceRegistry`; do **not** reuse 16/17 (taken) or renumber. **Knowledge capture is NOT a registry layer** (advisor-draft, event-triggered). So the registry goes 32→33 in WO-121 and is untouched by WO-123 — update the hard-coded `32`→`33` assertions in WO-121. The registry, not the spec appendix, is the operative layer numbering.
7. **Governance model per feature (WO-121 vs WO-123)** — default: **templates = admin learning-queue** (`learning_updates` candidate → `ApprovalFlow` → `TemplateImplementer` + ledger, because they're practice-wide); **knowledge capture = advisor-draft** (per-advisor draft → owning advisor accepts, no admin queue/ledger). Do not route knowledge drafts through admin approval.
8. **Offline dedupe transport + clear timing (WO-122)** — default: send the client `dedupeKey` as **`X-Idempotency-Key`** **and** the **queue-time `client_id`** as **`X-Portal-Client-Id`** on both sync `fetch` calls. The server keys the dedupe ledger on **`(user_id, queued client_id, operation/route, idempotency_key)`** + a **server-computed `request_fingerprint`** (file content / normalized body), and **stores the prior response**: same key+fingerprint → cached result; same key, different fingerprint → **409**; queued client not currently accessible → **403, keep**. (Don't key on the replay-time `resolveFor()` — it's `latest()` and can drift.) **Auth-flow redirects → JSON 401/419** for sync requests via an outer normalizer + exception `shouldRenderJsonWhen` (covers `auth`/CSRF/`mfa`/session-security/terms/verified). **Clear local queue + key on explicit logout only**; session-expiry keeps + retries.

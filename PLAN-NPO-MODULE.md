## 1. Non-negotiable principles (binding)

Everything in `CLAUDE.md` and the phase plans remains in force. The NPO-specific ones:

- **AI Integrity Principle (spec §3) — heightened for NPOs.** Every NPO AI output (dimension analysis, tension analysis, funder accountability narrative, impact-summary assist, cost/funding valuations) is **source-attributed, evidence-based, honest (no score inflation), and carries mandatory uncertainty disclosure**. All projections include a **±15% confidence range** (addendum §5.1/§5.2 — *no exceptions*). `AiClient` only; `FakeAiClient` in tests.
  - **Two distinct review regimes — don't conflate them.** **(i) FSA-generated AI analysis/narrative** (dimension findings, tension analysis, the funder *accountability* narrative, valuation framing) is a **hard advisor gate** — it **cannot reach a client or funder without explicit advisor approval; no auto-release, ever.** **(ii) Client-authored content with AI *language-assist only*** (the Impact Summary — the NPO's own document, no FSA IP, all facts entered by the client and **fact-checked** against platform data) uses an advisor review that **escalates via reminders and may auto-release on a configurable timer** (addendum §11.2). The auto-release is acceptable *only* because the artefact is the client's own words, not an FSA AI assertion, and the fact-check guard blocks unverifiable figures.
- **Mission inversion.** Scoring and recommendation logic must treat financial sustainability as instrumental to mission. Dimension 1 (Mission & Impact) is central; Dimension 3 (Governance) carries the **highest weight**; Dimension 7 (Compliance) is **critical**. The cost/funding valuations (§5) are framed in mission language, never as a commercial ROI ratio.
- **Learning governance — NPO layers are 34–37, NOT 33–36.** ⚠️ The addendum labels the new layers "33–36", but **Layer 33 is already occupied** by `TemplateSuggestionLayer` (shipped in WO-121); `LayerCadenceRegistry` currently holds **33** layers. Register the four NPO layers as **34 (Funder Database Updates), 35 (Governance Compliance Thresholds), 36 (Cost-Per-Beneficiary Benchmarks), 37 (Funding Concentration Risk Thresholds)** → registry total **37**. They emit **governed candidates only** (Detection → Summary Card → admin approval → 7-day notice → implementation → 30-day review → rollback). **Funder database changes flow through Layer 34 — never direct writes without admin approval.** Update the hard-coded count assertions (see WO-N18).
- **Funder Contact = report-scoped RLS (security-critical).** A funder sees **one** accountability report — *their* grant only. ⚠️ A `DdGuestLink`-style token reused as-is only carries **client**-level scope, and `reports`/`report_sections` RLS is **client-scoped** — so a funder token bound to the NPO's `client_id` would see **every report for that client**. The fix has three parts: **(1)** extend `fsa_set_request_context` (`RequestContext::apply`, `app/Support/RequestContext.php`) to also set a **`fsa_current_report_id`** GUC — **its value is derived *only* from the funder's validated (active, unexpired, unrevoked) link's bound `report_id`, never from a client-supplied id**, so a funder can only ever resolve to *their* link's report (not another `funder_accountability_report`); **(2)** because Postgres connections are **pooled/reused**, **every normal request must reset `fsa_current_report_id` to empty** (in `EnforceClientScope` alongside the existing client-scope reset) so stale report context can never carry over — a leak path otherwise; **(3)** funder sessions run under a **dedicated `funder_contact` role** with **no client-scope grant**, and each policy adds a funder clause keyed on the **right column**, **gated to the accountability report *type***, **and requiring the report be advisor-reviewed** (the existing `review_status = 'reviewed' AND reviewed_at IS NOT NULL` — so neither a wrong type nor a *draft* is reachable) — `reports`: `fsa_current_role()='funder_contact' AND reports.id::text = fsa_current_report_id() AND reports.type = 'funder_accountability_report' AND reports.review_status = 'reviewed' AND reports.reviewed_at IS NOT NULL`; **`report_sections`: `fsa_current_role()='funder_contact' AND EXISTS (SELECT 1 FROM reports r WHERE r.id = report_sections.report_id AND r.id::text = fsa_current_report_id() AND r.type = 'funder_accountability_report' AND r.review_status = 'reviewed' AND r.reviewed_at IS NOT NULL)`** (⚠️ `report_sections.id` is the *section* id — keying on it would show the report but **zero sections**). So a funder can't fall through to client-scoped visibility, and a normal role with an accidentally-set report id still can't use the funder clause. Enforced at the **DB layer, not just the UI** (addendum §10.2). Token-based (no full account, one-way `token_hash`), session-logged, 90-day expiry after grant end, advisor/NPO-revocable. Reuse `DdGuestLink` for the token/link mechanics only — *not* for scope.
- **ISA-2022 critical alert blocks downstream analysis.** If `legal_structure = incorporated_society` AND `isa_2022_reregistered = false`, generate a CRITICAL alert immediately and **pause/block other analysis until the advisor acknowledges** (addendum §3 Dimension 7, §15.3).
- **Audit + RLS everywhere.** Every new client-scoped table (`npo_engagements`, dimension data, funder records, drafts, sync/log tables) gets `ENABLE`+`FORCE` RLS + a `*_client_scope` policy; every mutation audited via `AuditWriter`; all external calls through `ResilientHttp` + feature flags (default OFF → fixture/fake). **Token storage — pick by recoverability:** *recoverable* third-party OAuth creds (calendar/accounting) go in **`KeyEnvelope`** and are added to `RewrapEnvelopes::sources()`; **bearer link tokens (funder contact) are stored as a one-way `token_hash`** (the `DdGuestLink` pattern — `char(64)`, never recoverable), **not** enveloped.
- **Reuse, don't rebuild.** SME engine (`BusinessHealthRadarBuilder`/`BusinessHealthSnapshotWriter`), the admin-managed questionnaire engine, `ProposalBuilder`/`SignoffFlow`, `ReportComposer`, Xero/MYOB/QuickBooks, the Companies Office integration, `DdGuestLink`, the WO-123 advisor-draft pattern, the Holidays Act checker — all are reused. NPO work is new *content + layers + config + reports*, not new infrastructure.

## 3. Cross-cutting data model (read before the WOs)

- **`EngagementType` enum** (`app/Enums/EngagementType.php`) currently has 4 cases. Add **`NPO = 'npo'`**. Audit every `match (EngagementType ...)` / exhaustive switch for a new arm (analysis runner, dashboards, reports, fee calc) — adding a case can break exhaustive matches; handle or default each.
- **NPO sub-type + config** live on a new **`npo_engagements`** table — **one row per engagement, NOT one per client** (a client can have a Governance Review row *and later* a full `standard_npo` row). Columns: `sub_type` (`standard_npo`/`social_enterprise`/`governance_review`), `legal_structure` (8 options, §2.1), **`tiriti_mode` (`standalone`/`woven`/`null` — these are the stored values; "Mode A"=standalone, "Mode B"=woven are display labels only, used consistently throughout)**, `social_enterprise` (bool), `commercial_weight`/`mission_weight` (integers summing to 100, nullable), `isa_2022_reregistered` (bool, nullable), **`converted_from_npo_engagement_id`** (nullable self-FK → the governance-review row this full engagement converted from), plus audit columns. RLS client-scoped. *(5a migration — WO-N01 — needs `sub_type`/`legal_structure`/`isa_2022_reregistered` **plus `converted_from_npo_engagement_id` and the conversion-status/3-year-trigger columns**, because WO-N07 (5a) creates the conversion record + full-advisory shell. The Te Tiriti/social-enterprise/weighting columns are added in 5b — WO-N08.)*
- **Governance-review lifecycle is its own row + a conversion record.** A `governance_review` `npo_engagements` row carries its **conversion status** (`report_delivered`/`declined`/`converted`), decline reason, report-delivery date, and the **3-year re-engagement trigger date** (on a sibling `governance_review_conversions` table or columns on the row — implementer's choice). On conversion, a **new** full `standard_npo` row is created with `converted_from_npo_engagement_id` pointing back (WO-N07, in 5a), and the governance findings are copied into the new row's Dimension 3 (WO-N09, in 5b — the copy needs the dimension model). **The conversion link + status schema therefore ships in 5a (WO-N01); only the Dimension-3 *copy* is deferred to 5b.** This keeps the standalone review's lifecycle independent of the full engagement.
- **Engagement identity on reused surfaces (multiple NPO engagements per client).** Because a client can hold a `governance_review` row *and* a later `standard_npo` row, the reused client-scoped surfaces — **`reports` (`client_id` only), `questionnaire_responses` (`client_id`), `proposals` (`client_id`)** — cannot tell *which* NPO engagement an artefact belongs to. **A single shared migration (WO-N01) adds a nullable, indexed `npo_engagement_id`** (null for SME so existing behaviour is unchanged — additive, not a Phase 1–4 modification) to `reports`, `questionnaire_responses`, `proposals`, **and `fee_calculations`**, each via a **composite FK `(npo_engagement_id, client_id) → npo_engagements(id, client_id)`** so an artefact can never be attached to a different client's engagement (the dual-scope integrity guard); each writer then **sets** it on creation — N02 (governance responses), N05 (governance report), N06 (governance proposal), N10 (questionnaire), N14 (accountability report + impact summary), N17 (reports). Lists/filters/RLS for NPO scope use it. *(Lighter alternative: a `metadata` jsonb key — queryable but unindexed; OD-10.)* This binds the governance-review evidence/report/proposal to the review engagement and the full-advisory ones to the converted engagement.
- **Dimensions** stored per engagement (`npo_dimension_scores`: dimension 1–8, score 0–100, advisor weight, findings jsonb, mode-B criteria contributions). Reuse the SME scoring/weighting + radar builder.
- **User roles:** `NPO Board Member` is a **new `User::TYPE_*`** (full read-only account, ≤10 per engagement) — but **`roleMatrix()` (coarse permission strings) is not sufficient** for its access rules. A board-member ↔ engagement link table carries a **`treasurer` boolean** (treasurers additionally see the Funding Strategy dimension). **Critically, board members get NO client scope** — if their `accessibleClientIds()` returned the client, client-scoped RLS would expose the *whole* client (all engagements/reports). Instead a board-member session sets **`fsa_current_npo_engagement_id`** — the requested engagement id may come from the route (a member may be on several engagements), but the server **validates it via `fsa_user_is_board_member_of()` before setting the GUC** (client-proposed, server-validated; reset each request, role-guarded `npo_board_member`) — and the RLS clauses also **prove membership** via the same `SECURITY DEFINER` helper (so the check doesn't recurse into `npo_board_members` RLS and fail closed), so a tampered GUC can't reach another engagement. The **`npo_engagements` row itself and `report_sections` get board clauses** (the former keyed on `id`, the latter via its parent report), and **`npo_board_members` is RLS-protected with an own-membership read** so board users can read their own links. **Fine-grained app policies** (report-type, report-section, financial-detail, treasurer) layer on top (WO-N15). DB-layer engagement scope + membership proof + app-layer granularity, all tested. `Funder Contact` is **token-based, NOT a `User::TYPE_*`** — model it on `DdGuestLink` for link/token mechanics, with report-scoped RLS for access (see §1). The addendum's "12 roles" counts the DD guest role + these two.

    - **The `npo_board_members` link table is security-critical — create + protect it explicitly.** Columns: `id`, `npo_engagement_id`, **`client_id`** (derived from the engagement at write time; together they form the **composite FK `(npo_engagement_id, client_id) → npo_engagements(id, client_id)`** so a membership can't reference a different client's engagement), `user_id`, `treasurer` (bool), `active` (bool), `revoked_at` (nullable), `created_by_user_id`, timestamps; unique `(npo_engagement_id, user_id)`. **RLS:** `ENABLE`+`FORCE` with a policy allowing **super_admin/system + advisors who manage the engagement (client-scoped via `client_id`) + the board member reading *their own* rows** (`user_id::text = fsa_current_user_id()`) — **without the own-row clause the membership check fails closed** (board members have no client scope).
    - **`SECURITY DEFINER` helper `fsa_user_is_board_member_of(engagement_id uuid) → boolean`** — every board RLS clause trusts it, so harden it: **fixed `SET search_path` (e.g. `pg_catalog, public`)**, **schema-qualified** table references (`public.npo_board_members`), **`REVOKE EXECUTE … FROM PUBLIC` + `GRANT` to the app DB role only**, **owned by a privileged role**, and explicit **`active = true AND revoked_at IS NULL`** semantics; it checks membership for `fsa_current_user_id()` **bypassing the table's RLS**, so the proof never recurses.
    - **DB-layer scope (not app-only):** board members return **empty `accessibleClientIds()`** (so client-scoped RLS never exposes the whole client); instead extend `fsa_set_request_context`/`RequestContext::apply` to carry **`fsa_current_npo_engagement_id`**. A board member may belong to **several** engagements, so the **requested engagement id may come from the route/request, but the setter MUST validate it via `fsa_user_is_board_member_of(requested_id)` before setting the GUC** (client-proposed, **server-validated** — reject/403 if not a member) — **reset on every normal request** (pooled-connection safety, same as the funder `report_id`). NPO-visible tables add a board clause that **proves membership at the DB layer via the helper**: `fsa_current_role()='npo_board_member' AND npo_engagement_id::text = fsa_current_npo_engagement_id() AND fsa_user_is_board_member_of(npo_engagement_id)` — so even a tampered GUC can't reach an engagement the user isn't linked to.
    - **`npo_engagements` itself needs a board clause** (it's client-scoped, but board members have no client scope, so they can't read their own engagement row otherwise): add `fsa_current_role()='npo_board_member' AND id = fsa_current_npo_engagement_id() AND fsa_user_is_board_member_of(id)`. This works directly for surfaces that **carry `npo_engagement_id`** — `reports` (via N01) and `npo_dimension_scores` (engagement-keyed by design). **`report_sections` has no `npo_engagement_id`**, so its board clause **joins to the parent report** (via the `SECURITY DEFINER` helper, so it doesn't recurse into `npo_board_members` RLS): `fsa_current_role()='npo_board_member' AND EXISTS (SELECT 1 FROM reports r WHERE r.id = report_sections.report_id AND r.npo_engagement_id::text = fsa_current_npo_engagement_id() AND fsa_user_is_board_member_of(r.npo_engagement_id))` — otherwise a board-allowed report renders **zero sections** (board members have no client scope). **⚠️ Other board-visible surfaces are still client-scoped and would leak across a client's engagements** — `milestones`, the action plan, uploaded minutes, board records: these must be **engagement-attributed in their owning WO** (`milestones`/action-plan in **N14**, minutes/board-records in **N16**) before the board clause can scope them, or be scoped via a join to an engagement-attributed parent. Do not expose a client-scoped board-visible surface without engagement attribution. **App-layer granularity on top** (`roleMatrix()` is coarse): a `ReportPolicy` gating by **report type** that is **deny-by-default with an explicit allow-list referencing the N01-added `ReportType` cases** (`governance_review_report`, `npo_health_report`) — so the Advisor report (and any type not on the list) is denied even before its template exists; a **report-section / financial-detail** gate (high-level summary only; full financials denied); and a **`treasurer` attribute** on the board-member↔engagement link that unlocks the Funding Strategy dimension. ≤10 per engagement, advisor-activated. **Build-order note:** the type *constants* exist from N01, so N15 references them cleanly; only the NPO Health + Advisor *templates/rows* arrive in **WO-N17** (after N15), so **N15's report-type tests cover what's instantiable by then** (Governance Review allowed; a non-allowed type denied by default); the **NPO Health (allowed) + Advisor (denied) gating tests are added in WO-N17** (where those report rows exist to test against).
  - **Funder contact:** a funder-link model on the `DdGuestLink` pattern — store a **one-way `char(64) token_hash`** (bearer link; **not** `KeyEnvelope`, which is for recoverable creds). Access is **report-scoped at the RLS layer** (see §1), with three required parts: (1) extend `fsa_set_request_context`/`RequestContext::apply` to carry **`fsa_current_report_id`** — **derived solely from the validated funder link's bound `report_id`** (the link is resolved by `token_hash`, checked active/unexpired/unrevoked, and its stored `report_id` is used; a client-supplied report id is ignored), so link A can never resolve to report B; (2) **`EnforceClientScope` (and any normal request) resets `fsa_current_report_id` to empty** so a pooled connection can't carry stale report scope; (3) funder sessions run under a dedicated **`funder_contact` role** (no client-scope grant); the `reports` policy adds `fsa_current_role()='funder_contact' AND reports.id::text = fsa_current_report_id() AND reports.type = 'funder_accountability_report' AND reports.review_status = 'reviewed' AND reports.reviewed_at IS NOT NULL` (the `ReportType` value + the **existing** review columns — not an invented `released_at`) and the **`report_sections` policy joins to its parent report on `report_id` and requires that report's `type = 'funder_accountability_report' AND review_status = 'reviewed' AND reviewed_at IS NOT NULL`** (the report FK — *not* `report_sections.id`, the section id). **Both the funder-link creation (invite flow) and the RLS clauses require the target report be a `funder_accountability_report` that is advisor-reviewed** — a link can never be minted against (nor a context resolve to) an Advisor / NPO Health / Impact Summary report, **nor a draft (un-reviewed) accountability report** (closing the path where a premature link would expose the un-approved AI narrative N14 hard-gates). Session logging, 90-day-post-grant expiry, advisor/NPO revocation; the NPO-initiated request → advisor approve/decline-with-reason → branded invite flow.
- **Frontend:** advisor activation of board members (≤10); the NPO-initiated funder-invite request + advisor approval queue; funder read-only report view.
- **Tests:** board member sees only the granted set (403 on full financials / Advisor Report / other clients); **a board member of engagement A sees zero reports/dimension rows of engagement B *of the same client* — at the RLS layer (board members have no client scope)**; **a board-allowed report renders its `report_sections` (positive), and a board member sees zero `report_sections` of a same-client *other-engagement* report (negative — proves the report_sections→reports join clause)**; **a board member requesting an engagement they're NOT a member of is rejected at GUC-set time (`fsa_user_is_board_member_of` returns false → 403), and even a tampered GUC is denied by the helper clause**; **a member of engagements A and B can switch between them (validated) but never reach C**; **a `npo_board_members` row whose `client_id` ≠ its engagement's client is rejected by the composite FK**; **a board member can read their own `npo_engagement` row and their own `npo_board_members` rows under RLS** (the membership check does not fail closed); **a revoked board member (`active=false`/`revoked_at` set) is denied everything** (`fsa_user_is_board_member_of` returns false); **a non-treasurer board member is denied the Funding Strategy dimension; a treasurer is allowed it**; **a funder session with its `report_id` context can `select` exactly one report, sees **all `report_sections` of *that* report** (positive — guards the `report_sections.report_id` clause), and zero `report_sections` of any other report — asserted at the query/RLS layer, including for another report of the *same* client** (the cardinal security test); **a funder context whose `report_id` points at a non-accountability report (Advisor / NPO Health / Impact Summary) sees nothing — and a funder link cannot be minted against such a report** (the report-type guard); **a funder link/context to a *draft* (un-reviewed: `review_status != 'reviewed'`) accountability report sees nothing — and a link cannot be minted against a draft** (the review guard); **funder link A cannot load accountability report B even when B's id is supplied — the GUC is derived from A's validated link, not the request** (the link-binding test); **a non-`funder_contact` role with `fsa_current_report_id` set still cannot use the funder clause** (role guard); **a normal request after a funder request has report context cleared** (no stale leak on a reused connection); session logged; access auto-expires at 90d; advisor + NPO can revoke; decline reason recorded.
- **Acceptance:** board members get scoped read-only access; a funder can only ever load their one report, enforced in the query.

## 6. Open decisions / assumptions (recommended defaults)

1. **OD-1 — WO namespace.** Default: **`WO-N01…N19`** (NPO namespace), matching the WO-D / WO-M track precedent. (Alternative: continue the global sequence WO-126+.)
2. **OD-2 — Layer IDs.** Default (binding correction): NPO layers **34–37**, not 33–36 (33 = `TemplateSuggestionLayer`). The addendum's numbers are conceptual; the registry is operative.
3. **OD-3 — `npo` engagement type vs sub-type.** Default: a single `EngagementType::NPO` with `sub_type` on `npo_engagements` (so existing 4-case matches gain one arm, not four). Governance Review is a sub-type, not a separate engagement type.
4. **OD-4 — Funder Contact modelling.** Default: `DdGuestLink`-pattern link (no `User` row), **one-way `token_hash`** (not `KeyEnvelope`); scope enforced by **report-scoped RLS** — `fsa_current_report_id` **derived from the validated link's bound report** (never client-supplied), **reset on every normal request**, honoured only for a dedicated **`funder_contact` role**, **and gated to `reports.type = 'funder_accountability_report'` AND the existing `review_status = 'reviewed' AND reviewed_at IS NOT NULL`** (so neither a wrong report id, a wrong type, nor a *draft/un-reviewed* report is reachable). Not the client-scoped token alone. (Matches §10.2 "No Full Account" + the §1 isolation requirement.)
5. **OD-5 — Dimension-3 pre-population timing.** Default: copy governance findings into Dimension 3 **at conversion** (WO-N07 creates the shell; WO-N09 performs the copy) — requires 5a's data model frozen before 5b (addendum §15.2).
6. **OD-6 — Te Tiriti storage + vocabulary (resolves Open Q1).** Default: **store `tiriti_mode = standalone | woven`** (Mode A = standalone, Mode B = woven are *display labels only* — used consistently). In `woven`, `[TIRITI]`-tagged criteria are rows within their parent dimension contributing to its score; no separate Dimension 8 row exists.
7. **OD-7 — Funder DB seed.** Default: ship structure + Layer-34 governance + a *small fixture set*; real records are a Super-Admin content task at launch (addendum §15.2), not in these WOs.
8. **OD-8 — Blended SE score (resolves Open Q2).** Default: weights stored as **integers summing to 100**; `blended = (commercial_score*commercial_weight + mission_score*mission_weight) / 100` (a weighted average on the 0–100 scale). The `/100` is mandatory; update the CLAUDE.md block (WO-N19) to match.
9. **OD-9 — Conversion linkage.** Default: each engagement is its own `npo_engagements` row; the full advisory row links back via `converted_from_npo_engagement_id`; conversion status + 3-year trigger live on the governance-review row (or a `governance_review_conversions` sibling). No 1:1-per-client assumption.
10. **OD-10 — Engagement identity on reused surfaces.** Default: **one shared migration in WO-N01** adds a **nullable, indexed `npo_engagement_id`** to `reports`, `questionnaire_responses`, `proposals`, **and `fee_calculations`** (null for SME — additive, no behaviour change), each via a **composite FK `(npo_engagement_id, client_id) → npo_engagements(id, client_id)`** (OD-12), **and replaces the `questionnaire_responses` uniqueness with two partial indexes** (legacy `WHERE npo_engagement_id IS NULL` + NPO `WHERE npo_engagement_id IS NOT NULL`). Later board-visible surfaces (`milestones`/action plan in N14; minutes/board records in N16) get the same `npo_engagement_id` + composite FK. Every writer then only **sets** it: N02 (governance responses), N05 (governance report), N06 (governance proposal) in 5a; N10 (questionnaire), **N14 (Funder Accountability Report + Impact Summary)**, N17 (reports) in 5b. (Alternative: a `metadata` jsonb key — no schema change but unindexed; rejected because NPO scope needs to filter/RLS on it.)
11. **OD-11 — Governance findings store.** Default: a dedicated **`governance_review_findings`** table keyed by `npo_engagement_id` (RLS client-scoped) — N04 writes, N09 copies/backfills by engagement id. (Alternative: add `npo_engagement_id` to the existing client-scoped `analysis_findings`; rejected as the default to avoid widening a shared Phase-2 table and to keep repeat-review findings cleanly separated.)
12. **OD-12 — Same-client integrity guard.** Default: enforce `row.client_id = npo_engagements.client_id` via a **composite FK `(npo_engagement_id, client_id) → npo_engagements(id, client_id)`** on every engagement-keyed surface (`reports`, `questionnaire_responses`, `proposals`, `fee_calculations`, `governance_review_findings`, + the board surfaces attributed in N14 (`milestones`, action plan) and N16 (minutes, board records)), backed by `UNIQUE(id, client_id)` on `npo_engagements`. (Alternative: a trigger/`CHECK`; rejected — the composite FK is declarative and free.)

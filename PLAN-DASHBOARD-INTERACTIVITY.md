## 0. How to use this plan

1. Read §1–§5 once before starting. They carry forward the non-negotiable principles, record the **current-state audit** (what exists vs. what's net-new — this is the "huge implications" map), and lock the cross-cutting UI foundations every widget WO reuses.
2. Implement **Work Orders** in §6. **WO-D01 (foundations) blocks everything.** After it, WO-D02…WO-D08 are largely independent and may be done in any order; WO-D09 (QA) is last.
3. **Commit cadence (matches Phase 3/4 §0 — this supersedes the older "one WO per branch/PR" rule in `CLAUDE.md`):** single `featureApp` branch, **no WO branches/PRs**; **one commit per WO** (`WO-D0x: <slug>`); **`IMPLEMENTATION.md` updated in the same commit**; full verification + push **at the track boundary** (after WO-D09). Regenerate Wayfinder (`php artisan wayfinder:generate --with-form`) on any controller/route change.
4. Every WO ships with its tests. A WO is done only when its acceptance criteria are demonstrably true and the gate is green: **the PHPUnit suite via `php -d memory_limit=1024M vendor/phpunit/phpunit/phpunit --no-coverage`** (⚠️ **not** `composer test` — that script runs `@php artisan test`, which **hangs in this environment**; the direct PHPUnit binary is the working command, and the default 128M `memory_limit` OOMs on a full run), plus `npm run lint:check`, `npm run types:check`, `npm run format:check`. (WO-D09 optionally repoints the `composer test` / `ci:check` scripts at the direct binary so the named gate works again.)
5. **This track adds zero new spec scope.** Every widget visualises data the platform already produces (or, for the two net-new Tier-1 widgets, aggregates data that already exists). If a widget would require inventing a new analysis/behaviour, stop and raise it — do not build it.
6. **The trigger previews; an explicit "Open ›" action drills.** Touch devices have no hover, and on touch a single tap cannot both open a preview *and* navigate. So the contract (resolved in §4.1) is: the trigger's primary interaction only ever **opens the preview** (hover on a mouse, focus on the keyboard, tap on touch); when a card is drillable, the **drill is a separate, focusable "Open ›" action inside the preview** — the one consistent drill path on every pointer type. A card may also be **preview-only (no `drillHref`, no "Open ›")** for aggregate/unmapped elements. The shared `InsightHoverCard` (WO-D01) enforces this so no widget reinvents it.

## 1. Non-negotiable principles (carried forward — still binding)

Everything in `CLAUDE.md`, `PLAN.md` §1, and the Phase 2–4 §1 sections remains in force. This track stresses these hardest:

- **AI Integrity Principle (spec §3).** Every hover that surfaces an AI-derived number (engagement score, red-flag trigger metric, radar dimension score, PV assumption) must be **evidence-based and source-attributed** — the hover shows *what drove the number*, not an unexplained figure. Red-flag and radar hovers cite the underlying `analysis_finding.attributions` (that is the evidence column on `analysis_findings`; `source_attributions` is the column name used by opportunities/risks/pv). Low scores are stated plainly (kindness in delivery, never in content). **No fabricated context** — if a data point isn't available, the hover says so; it never invents a plausible value.
- **No silent learning (spec §3, §23).** Nothing in this track modifies scoring weights, prompts, or platform behaviour. Engagement/radar scoring formulas are **fixed config** (`config/dashboards.php`) — they are not learning layers and never self-tune. Tunable thresholds are owner-set constants, not auto-adjusted.
- **Security baseline (spec §4).** All client-scoped reads stay behind RLS + `EnforceClientScope` (the dashboards already run scoped). The **Payments** widget (WO-D07) touches the most sensitive surface — **no card data, no PAN, no full gateway payloads in any payload or log line**; surface only `amount`, `failed_reason` (already a sanitised string), `attempt`, dates. All log lines through the `Redactor`. Every drill action that mutates (payment retry, red-flag acknowledge) stays audited via `AuditWriter` and permission-gated.
- **No PII in raw logs.** Hover/drill endpoints log via the redaction helper. NZBN is not masked (existing rule); names/emails are.
- **Tests use `FakeAiClient`.** No widget computation calls a live AI path.

## 4. Cross-cutting foundations (built in WO-D01, reused by all)

### 4.1 `InsightHoverCard` — the one hover/drill primitive
A single wrapper at `resources/js/components/insight/InsightHoverCard.tsx` so no widget reinvents hover UX or accessibility. **The trigger only previews; it never navigates.** When a card is drillable, the drill is a focusable **"Open ›"** action inside the preview — the same affordance on every pointer type, removing the touch tap/navigate conflict.
- **Fine pointer (mouse)** → Radix **HoverCard** opens on hover (`openDelay≈150ms`). Hovering shows the preview; the user clicks the in-card **"Open ›"** link to drill.
- **Keyboard** → focusing the trigger opens the preview; the **"Open ›"** link (when present) is in the tab order and Enter/Space activates it.
- **Coarse pointer (touch)** → Radix **Popover** opens on tap (detected via `matchMedia('(hover: none)')`, `usePointerCoarse()`). Tapping the trigger opens the preview only — it **never** navigates; the user taps **"Open ›"** to drill.
- **Props:** `title`, `rows` (label → value pairs, optional ↑↓ delta colouring); **optional `drillHref`** (the "Open ›" target — a real `<Link>`/anchor). **`drillHref` is optional, not required:** when set the card renders the "Open ›" control; **when omitted the card is an explicit preview-only / non-drillable state with no "Open ›"** — this is the intended mode for aggregate or unmapped elements (e.g. the PV "Other …" remainder bar, or a PV bar whose row has no `analysis_finding_id`; WO-D04). Optional `footer` for source attribution. `aria-describedby` wires the card content to the trigger; the "Open ›" link (when present) carries an `aria-label` naming the destination.
- **Tested invariant:** the trigger has no navigation side-effect; the **only** drill path is the `drillHref` "Open ›" control when supplied, and a card without `drillHref` renders no navigating control at all. (Removes the line-95/96 ambiguity *and* the contract/non-drillable-bar conflict.)

### 4.2 `useDrillFocus` — anchor + highlight drill convention
- A query-param contract: drilling to a page passes `?focus=<section-key>` (and optional `&highlight=<element-id>`), e.g. client page `…/advisor/clients/{id}?focus=analysis&highlight=<finding-id>`, the portal radar `…/portal?focus=health&highlight=health-<dimension>`, or filtered list `…/advisor/clients?exposed_to=ocr`. **`highlight` targets any stable element id** — a record id (a finding) **or** a section sub-anchor (a radar dimension like `health-financial`); there is no separate `dimension` param.
- A hook `useDrillFocus()` (in `resources/js/hooks/`) reads `focus`/`highlight` on mount and scrolls/highlights, with an **explicit id-resolution contract** (resolves the section-key vs anchor-id mismatch): **`focus=<key>` resolves to the element `id="section-<key>"`** (so `?focus=goals` → `#section-goals`), and **`highlight=<id>` is used as the *raw* element id** (no `section-` prefix — e.g. a finding id, or a sub-anchor like `health-financial`). The hook `scrollIntoView({behavior:'smooth'})` to `highlight`'s element if present, else to `section-<focus>`, and applies a transient highlight ring (~2s). Widgets that drill to a sub-element render a stable raw id for it (e.g. the per-dimension findings blocks render `id="health-<dimension>"`).
- WO-D01 adds **stable `id` anchors** to the drill-target sections in `advisor/clients/Show.tsx` (`id="section-overview"`, `id="section-analysis"`, `id="section-goals"`, `id="section-payments"`, **and the engagement-drill targets `id="section-questionnaire"`, `id="section-documents"`, `id="section-messages"`** — see WO-D02's component→key map) and `portal/Dashboard.tsx` (`id="section-health"`, per-dimension `id="health-<dimension>"`). Wayfinder route helpers carry the query params.
- **Highlighted-entity guarantee (server side):** a `highlight=<id>` drill is useless if the destination payload doesn't include that row. Several destination payloads are **capped** (e.g. `ClientController::analysisFindingSummaries()` limits findings to **20**, so an older flagged/PV-sourced finding can be absent). The rule for every drill target: **when `highlight=<id>` is present, the destination controller must ensure that specific row is in the payload** — merge it in even if it falls outside the default cap (load the highlighted id explicitly and union it with the capped list). Each widget WO that drills to a capped list states this; WO-D01 establishes it as the shared convention.

### 4.3 Config
`config/dashboards.php` (new) holds the **fixed** tunables (not learning-governed): engagement-score weights + G/A/R thresholds, radar dimension→module map + score thresholds, economic-exposure rules. Documented defaults in §7; owner-overridable via env where it makes sense.

### 4.4 Performance / Inertia
Hover content must be **already in the page payload** — hovering must not fire a request. Each widget WO extends its existing Inertia payload with the hover fields (the panels already ship their data up front). Where a payload would balloon (e.g. per-client funnel detail, per-recommendation PV), keep it bounded (top-N, as the panels already do) and lazy nothing on hover. Drill navigations are normal Inertia visits.

## 5. Database / schema additions (small)

Conventions unchanged: `uuid` PKs via `gen_random_uuid()`, `jsonb`, `client_id` + **RLS** (`ENABLE`+`FORCE` + `*_client_scope` policy mirroring the existing tables) on any client-scoped table, audit via `AuditWriter`.

- **`red_flags.trigger`** *(nullable `jsonb`, WO-D03)* — optional structured "what tripped this" `{metric, value, comparison, window}`. Backfill/derive from the linked `analysis_finding.attributions` (⚠️ the `analysis_findings` evidence column is `attributions`, **not** `source_attributions` — that name belongs to `improvement_opportunities`/`risk_costs`/`pv_calculations`) where present; new red flags populate it at creation. If derivation from existing findings is clean enough, this column may be skipped — decide in WO-D03 (see §7).
- **`business_health_snapshots`** *(WO-D08)* — `id`, `client_id` (RLS), **`assessment_batch_id` (uuid)**, `dimension` (financial|operational|people|strategic|compliance), `score` (0–100, **nullable** — null = "insufficient data", see WO-D08), `top_finding_id` (→`analysis_findings`, nullable — the headline finding for the hover), **`contributing_finding_ids` (jsonb — every finding that fed the score)**, **`module_run_states` (jsonb — keyed by `AnalysisModule`; each value is the object `{ state: scored|completed_no_findings|completed_no_client_safe_findings|never_run|in_progress|blocked|failed, scoring_run_id: uuid|null (the latest *completed* run used for the score), scoring_completed_at: iso8601|null (that run's `analysis_runs.completed_at`), latest_run_id: uuid|null, latest_run_status: string|null, latest_run_at: iso8601|null, stale: bool (latest run newer than the scoring run and not completed) }` — **persist the timestamps too** so the stale-run hover copy ("score reflects the last completed analysis on <date>; a newer run is <status>") renders from the snapshot **without re-querying live `analysis_runs`**; the migration/model/tests store this whole shape, not just compute it transiently)**, **`dimension_run_state` (string — derived rollup via a single **deterministic precedence** (first match wins, so the single string is unambiguous even when modules disagree): 1) `scored` if any mapped module is `scored`; 2) `blocked` if any is `blocked`; 3) `failed` if any is `failed` (**blocked beats failed** — blocked is client-actionable: upload docs / fix data quality); 4) `in_progress` if any is queued/running; 5) `completed_no_client_safe_findings` if any completed with findings but none client-safe; 6) `completed_no_findings` if any completed truly empty; 7) `never_run`)**, `captured_at`, `source_attributions` (jsonb — **aggregated across all contributing findings, not just the top one**). Stores the per-dimension rollup so the radar shows **trend vs last assessment** without recomputing history.
  - **Per-module provenance (a dimension maps to several modules):** e.g. `strategic` → `Swot`+`Competitor`+`WebsiteAudit` (§7.4), so a single dimension-level flag can't represent **partial coverage** (Swot scored, Competitor never run, WebsiteAudit ran-empty). Store the state **per module** in `module_run_states` and derive `dimension_run_state` from it. The hover for a partially-covered scored dimension can note which modules contributed vs. weren't run.
  - **Null-score axes still carry provenance (resolves the no-finding gap):** a zero-finding dimension has no `top_finding_id`/`contributing_finding_ids`/`source_attributions`, so finding-based provenance alone can't tell a module that **never ran** from one that **ran and produced no persisted findings**, nor from one that is **blocked/failed/in-progress.** `module_run_states` (with run ids) records that at snapshot time, so a later portal read renders the honest distinction without re-querying live runs.
    - **⚠️ Two different runs can matter — define precedence (resolves the latest-vs-latest-completed conflict).** A module can have an **older `completed`** run (the one with a usable result) **and** a **newer `failed`/`blocked`/`running`** run. The score **always comes from the latest `completed` run** (the only run with a result); the per-module entry stores both: a **state** derived from that scoring run plus **`scoring_run_id`**/**`scoring_completed_at`**, **and** the **latest-overall run** (`latest_run_id`/`latest_run_status`/`latest_run_at`) with a **`stale` (bool)** flag. **Deterministic run ordering with a stable tie-breaker (timestamps alone can tie):** "latest run" and `latest_run_at` are ordered by **`COALESCE(completed_at, started_at, created_at)` desc, then `id` desc**; the scoring run is the latest `status = completed` run ordered by **`completed_at` desc, then `id` desc**. ⚠️ `id` is a **random `gen_random_uuid()`**, so the `id` tie-break is an **arbitrary-but-stable** disambiguator (it makes the choice deterministic across executions/tests) — **not** a chronological signal. A genuine tie on all of `completed_at`/`started_at`/`created_at` means the runs are effectively simultaneous, so "newer" is undefined anyway; the tie-break just picks one consistently. **`stale = (latest_run_id != scoring_run_id) AND latest_run_status != completed AND COALESCE(latest.completed_at, latest.started_at, latest.created_at) > scoring.completed_at`** — note the **strict `>`**: at an exact-instant tie the newer run is **not** considered strictly newer, so `stale = false` (no special "non-completed wins the tie" rule, which the random-UUID comparator couldn't guarantee anyway). `latest_run_*` is still whatever the deterministic comparator selects; `stale` simply requires the latest non-completed run to be **strictly after** the scoring run. So a module shows `scored` from its completed run **and** flags "a newer analysis is in progress / blocked / failed." If there is **no** completed run at all, the state is taken from the latest run: `queued`/`running` → `in_progress`; `blocked_documents`/`blocked_data_quality` → `blocked`; `failed` → `failed`; no run row → `never_run`.
    - **Scored vs the two "empty completed" states (resolves the mislabel):** a completed scoring run maps to `scored` if it has ≥1 **client-safe** finding (lens ≠ `Prescriptive`); to **`completed_no_client_safe_findings`** if it produced findings but **none client-safe** (e.g. **Prescriptive-only** — recommendations exist, they're just not radar-visible); and to **`completed_no_findings`** only if it produced **zero** persisted findings at all. These are distinct so the copy is never false (see hover copy below). ⚠️ Without these distinctions a Prescriptive-only run would wrongly read "no persisted cited findings," and a blocked/failed module would read "no analysis run yet."
    - **Hover copy (per state — none false):** `scored` → score shown; `completed_no_findings` → "Analysis completed with no persisted cited findings"; **`completed_no_client_safe_findings`** → **"No client-safe findings to display for this dimension"** (recommendations may exist but aren't shown on the radar — **not** "no findings"); `blocked` → "analysis blocked — resolve document / data-quality issues"; `failed` → "analysis didn't complete — rerun needed"; `in_progress` → "analysis in progress"; **`stale`** appends "(score reflects the last completed analysis on `scoring_completed_at`; a newer run is `latest_run_status`)".
  - **Evidence must cover the whole claim (AI-integrity):** the `score` is an aggregate severity-load over **all *client-safe* contributing findings in the window** (lens ≠ `Prescriptive`; per the WO-D08 client-safety filter — the window is client-safe *before* scoring), so the citation can't be a single row. Persist **`contributing_finding_ids`** (the full **client-safe** set) and aggregate **`source_attributions`** from every contributing finding's `attributions`; `top_finding_id` is only the most-severe one surfaced in the hover headline. ⚠️ "All findings" here always means the **client-safe** set — a `Prescriptive` finding never enters `contributing_finding_ids` or `source_attributions`. (If the owner prefers a simpler model, the alternative is to make the score explicitly *top-finding-only* — but then the rollup must score on that one client-safe finding, not the set. Default = full client-safe-set score + full client-safe-set evidence.)
  - **Completeness is enforced in the schema + write, not just by convention (resolves the "convention only" gap):** add a **`CHECK` constraint restricting `dimension` to exactly the five valid values** (`financial`, `operational`, `people`, `strategic`, `compliance`) — so a typo/unknown dimension can't be inserted — plus a **unique key `(client_id, assessment_batch_id, dimension)`** (at most one row per dimension — no duplicate axes). All **5 dimension rows are written in a single DB transaction**, so a batch is atomically all-or-nothing — a partial batch can never persist. **"Latest complete batch" selection rule:** pick the most recent `assessment_batch_id` ordered by **`captured_at` desc, `assessment_batch_id` desc** (the batch id is a **stable tie-breaker** so two batches sharing a `captured_at` always resolve to the same one across DB executions/tests — never nondeterministic) that has **exactly the five expected dimensions present** — `WHERE dimension IN (the five) … HAVING count(DISTINCT dimension) = 5` (the `CHECK` + this exact-set filter mean "5 distinct" can only be the valid set, not five typos); defensively ignore any batch that isn't complete. **Trend = that latest complete batch vs the previous complete batch** (whole-assessment to whole-assessment), so axes never compare mixed dates or read a duplicate/partial batch.
  - **Why a batch boundary (resolves the partial-snapshot risk):** `App\Services\Analysis\AnalysisRunner` completes **one module at a time** and `analysis_runs` carry **no assessment-cycle id**, so hooking "on analysis-run completion" would write mixed-date, partial five-axis snapshots and a misleading trend, and there's no reliable signal for "a full cycle finished." Instead a snapshot batch is materialised **only on an explicit recompute trigger / backfill** (never auto-hooked to a module run) — see WO-D08.
- **No other schema changes.** Engagement score, economic deltas/exposure, PV per-recommendation, and funnel per-client are all **computed at read time** from existing tables.

### WO-D02 — #1 Client Engagement Score (advisor)
- **Goal:** Replace the data-quality-only badge in the client health table with a **4-component G/A/R engagement score** with a hover breakdown and a drill to the weak component.
- **Depends on:** WO-D01.
- **Backend:** new `app/Services/Dashboards/ClientEngagementScorer.php` computing per client. **Denominators are defined against the real schema (there is no "assignment" table and no "expected document categories" registry):**
  - **`questionnaire_pct`** — over the client's existing `questionnaire_responses` rows (a response row is the only "assigned" signal; **no assignments table exists**). **Reuse the existing completeness logic, do not reinvent it:** count only **conditionally-visible** questions via `QuestionnaireRuleEngine::visibleQuestionIds($questionnaire, $answers)` and count an answer as present via `answerHasValue($question, $answer)` — exactly as `DataQualityScorer` does (lines ~90/103/295). A raw "`required` + non-null `value`" count would over-count hidden questions and mis-read empty-but-present answers, producing false low scores.   - **`documents_pct`** — **reuse the existing `DataQualityScorer` verified-documents signal** (`verified clean documents / total uploaded documents`), since the platform tracks *verified uploaded* documents, **not** an "expected categories" checklist. If any blocking verification flag exists, treat the component as 0 (mirrors `DataQualityScorer`). Do **not** invent an expected-category denominator.
  - **Milestones** — the **score component is `milestones_on_track_pct`** (0–100, higher = better): `milestones_on_track_pct = total_active == 0 ? 100 : clamp(0, 100, round(100 × (total_active − off_track) / total_active))`, where `total_active` = milestones not `completed` and **`off_track` = active milestones past `due_date` OR `status = blocked`** (⚠️ `Milestone::STATUS_BLOCKED` exists — a future-dated **blocked** milestone is *not* overdue but is clearly not on track, so it counts as off-track). **No milestones → 100** (vacuously on-track — fixed default; alternative: drop the component and reweight, §7.1). The raw **display fields** `overdue_count` + `blocked_count` are returned separately for the hover.
  - **Comms** — the **score component is `comms_recency_pct`** (0–100, higher = better): `comms_recency_pct = clamp(0, 100, round(100 × (1 − days / config('dashboards.engagement.comms_decay_days'))))` where `days = now − message_threads.last_activity_at`, default `comms_decay_days = 30` (0 days → 100, ≥30 → 0); **null/never-messaged → 0**. The raw **display field** `last_comms_days` is returned separately for the hover.
  - **⚠️ Score keys are distinct from display fields (resolves the mixed-identifier hazard).** The composite and the **weakest-component** comparison operate **only** over the four **0–100, higher-is-better score keys** — `questionnaire_pct`, `documents_pct`, `milestones_on_track_pct`, `comms_recency_pct` — so `weakest_component = argmin` over those four is well-defined (never compares a raw inverted `last_comms_days` or a count). Raw **display fields** (`overdue_count`, `blocked_count`, `last_comms_days`) are carried alongside **for hover copy only** and never enter the score or the weakest pick.
  - Weighted composite `score = round(Σ weight × score_component)` — **rounded half-up to an integer 0–100** (the weighted sum is fractional, so the precision must be fixed or boundary cases like 74.5 diverge between implementations; round half-up means 74.5 → 75). The integer `score` is then banded into `level` (green/amber/red) using `config('dashboards.engagement')` thresholds (§7.1; bands are evaluated on the rounded integer). Deterministic for fixed fixtures so tests assert exact scores **and** the boundary (e.g. a fixture summing to 74.5 → score 75 → green).
  - Return `{ level, score, scores: { questionnaire_pct, documents_pct, milestones_on_track_pct, comms_recency_pct }, display: { overdue_count, blocked_count, last_comms_days }, weakest_component }` (where `weakest_component` is one of the four score keys). Compute in batch for the ≤20 clients the health panel already loads (no N+1 — reuse the existing aggregate-then-map pattern in `clientsHealth()`).

The scorer (or the frontend) maps `weakest_component → focus section-key`: `questionnaire_pct → questionnaire`, `documents_pct → documents`, `milestones_on_track_pct → goals`, `comms_recency_pct → messages`. ## 7. Open decisions / assumptions (recommended defaults — owner may override)

Codex should proceed with these defaults and flag them in the WO commit so the owner can tune:

1. **Engagement score (WO-D02)** — weights: questionnaire 30% · documents 30% · milestones-on-track 25% · comms-recency 15% (sum 1.0). The composite is **rounded half-up to an integer 0–100** before banding. Thresholds (on the rounded integer): **green ≥ 75**, **amber 50–74**, **red < 50**. Each component is a 0–100 sub-score (see WO-D02): **milestones = on-track %**, **no milestones → 100** (alternative: drop the milestone component and reweight the other three to 35/35/30 — owner choice); **comms decays linearly to 0 by `comms_decay_days = 30`**, never-messaged → 0. *(All config-driven in `config('dashboards.engagement')`; not learning-governed.)*
2. **Red-flag trigger (WO-D03)** — prefer deriving from `analysis_finding.attributions` (the evidence column on `analysis_findings`); only add the `red_flags.trigger` column if existing findings don't carry a clean metric/value/window. Never fabricate a trigger — omit with "source unavailable" if absent.
3. **PV per-recommendation cap (WO-D04)** — show top 8 by `rank` per client as individual bars; aggregate ranks 9+ into one non-drillable **"Other …" remainder step** so the displayed steps still sum exactly to the summary totals (no dropped value).
4. **Radar scoring (WO-D08) — exact, fixed formula (so tests assert exact scores, not just 0–100 bounds).** ⚠️ `analysis_findings` has **no open/closed state** (only `severity`/`lens`/`title`/etc.), so "open findings" is undefined — the window is **every client-safe finding (lens ≠ `Prescriptive`; "client-safe" defined concretely in WO-D08) from the most recent `status = completed` run per mapped module**, weighted by `severity` (`FindingSeverity` = `info`/`low`/`medium`/`high`/`critical`). All values are **fixed `config('dashboards.radar')`**, owner-tunable but versioned with the plan:
   - **`severity_weights`** (default): `info = 0`, `low = 1`, `medium = 3`, `high = 7`, `critical = 15`.
   - **`load`** for a dimension = `Σ severity_weights[finding.severity]` over its contributing findings.
   - **`load_cap`** (default `30`) = the load at which the score floors to 0.
   - **`score = clamp(0, 100, 100 − round(100 × load / load_cap))`** — deterministic integer. Worked examples (cap 30): one `high` (7) → `100 − round(23.3) = 77`; `high + medium` (10) → `100 − round(33.3) = 67`; two `critical` (30) → `0`; zero findings → **null** (insufficient data, not 100).
   - **Trend** = `score` delta vs the most-recent prior complete batch (up/down/flat); first batch → "baseline, no trend"; a null score is skipped (no trend arrow).
   - Tests assert these exact numbers for fixed fixtures.
   - **Default `config('dashboards.radar')` dimension → `AnalysisModule` map** (use real enum cases — there is no `Strategic` module; owner-tunable):
     - `financial` → [`Financial`]
     - `operational` → [`Operational`, `Systems`]
     - `people` → [`Hr`]
     - `strategic` → [`Swot`, `Competitor`, `WebsiteAudit`]
     - `compliance` → [`Compliance`, `RegulatoryImpact`, `InsuranceRisk`]
   - **Excluded from the radar by default** (not business-health dimensions): `KnowledgeAssessment` (advisor input), `Scenario`/`Succession` (forward-looking), `DdWorkstream` (separate DD engagement). Owner may remap.
5. **Economic exposure (WO-D05)** — rules in `config/dashboards.php`, but **only those whose backing data exists today emit a count** (see §7.12 for the availability verdict): **CPI → all active** (supported); **OCR → clients with debt/financing derivable from latest `financial_snapshots`** (supported iff derivable); **wage-by-industry** and **FX-by-import/export** are **unsupported** (no `clients` industry/trade fields) and render **"exposure unavailable"** — no count, no drill. Never fabricate a count from absent classification.
6. **Funnel drill (WO-D06)** — inline-expand the dropped-client list if ≤ ~10, else drill to a filtered clients list. Implementer's choice based on payload size.
7. **Radar split (WO-D08)** — split into D08a/D08b only if needed to keep each commit green.
8. **Manual payment retry cap (WO-D07)** — default: a manual Retry is a **one-shot advisor override that runs one attempt even past `max_attempts`** (the cap governs only the automatic scheduler). Alternative the owner may pick: **block** manual retry once `attempt >= max_attempts` (terminal). One-line guard either way.
9. **No-finding radar dimension (WO-D08)** — a dimension with zero findings renders as **"insufficient data" / unavailable** (not a fabricated score), cited to the absence of findings / latest module-run state. See WO-D08.
10. **Manual-retry success reactivation (WO-D07)** — default: a **successful** manual retry on a **paused recurring** schedule **reactivates** it (status → active + advance `next_run_at`) so auto-billing resumes. Alternative: keep it manual-only and require a separate explicit reactivation. (`advanceSchedule()` alone does not clear `paused`, so this must be explicit.)
11. **Radar drill vs preview-only (WO-D08)** — default: **build** the client-safe in-page findings section so scored axes drill to it. Fallback if descoped: scored axes are **preview-only** (no Open ›). Either way, unavailable/blocked/failed/in-progress axes are always preview-only.
12. **Economic exposure data availability (WO-D05)** — only rules whose backing data **exists today** produce a count: **CPI → all active clients** (supported); **OCR → clients with interest-bearing debt** read via **explicit configured JSON paths** into the latest `financial_snapshots` (the `balance_sheet`/`cash_flow`/`metrics` columns are arbitrary jsonb), using a **three-way exposed / not-exposed / `unknown`** result — a missing snapshot or missing configured key is `unknown` and excluded from both counts, **never silently "not exposed."** **Wage-by-industry and FX-by-import/export are NOT supported** — `clients` stores no industry/import/export classification and NZBN population doesn't persist it — so those indicators render **"exposure unavailable (classification not captured)"** with **no count and no drill**, never a misleading number. Adding `clients.industry_code` / trade flags, or named snapshot debt columns, is a future schema change, out of scope here.
